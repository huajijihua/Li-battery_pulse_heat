## 3. M3 建模定位

M3 是通用、模块化的低温脉冲加热**系统集成设计与参数匹配模型**。它应在系统边界内表达电池、高压配电与 DC-link、逆变器、PMSM、控制器、热管理、保护逻辑和必要的简化机械约束，并以工程兼容性筛查作为验证方法。

M3 服务于：

1. 通用低温脉冲加热系统集成设计和参数匹配。
2. 跨子系统电压、电流、功率、热、频率、时序和保护约束的兼容性判断。
3. 能量路径、热源分配、参数敏感性和控制策略的可审计分析。

M3 不服务于：

1. 未经数据验证的具体产品定型、析锂安全、寿命、NVH、退磁或量产许可结论。
2. 完整 CAD、壳体、支架、线束三维布局或结构有限元建模。
3. 直接继承历史车型、L0.5 或 M2 的默认参数和架构。

## 10. 并行计算工作流（按需开启）

并行计算不是默认行为，按需开启：仅在多工况独立扫描确实受益时启用，不对不适合的任务强行并行。

适合并行：多工况独立扫描（如 M4-02B 的 26 工况、M4-03 控制策略比较），各工况无数据依赖、各自构造 `SimulationInput` 非破坏性参数注入；工况数足够多（经验上 ≥ 4）且单工况分钟级，并行收益能覆盖池启动与模型编译开销。

不适合并行（保持串行）：单工况或极少工况；工况间有顺序/数据依赖；诊断单工况（`diagnose:XX`）、preflight 等只需一两次仿真的任务；并行会改变结果语义或破坏账本可追溯性的场合。

受控运行器统一遵循的并行约定：

1. 用 `parsim`，不用 `parfor`+`sim`（MATLAB 代码分析器官方建议；自动处理 worker 模型加载、`ShowProgress` 进度、错误捕获）。
2. 池规模保守：默认 `2` 个 worker，硬上限 `4`（请求超过 4 截断），由运行器内 `resolveWorkerCount` 统一控制；不按 CPU 核心数全开，保持机器响应、规避内存与许可压力。
3. 并行适配只在脚本侧：`SimulationInput` 与串行路径完全一致（仅 `applyScenario`），**不为并行修改模型本体**；FastRestart 等由 `parsim` 选项（`UseFastRestart`）在脚本侧控制。
4. 脚本侧 `addpath` 模型目录，让 `parsim` 把 `.slx` 传到 worker。
5. 保留串行回退：`runXxx(...,1)` 走原串行分支，不启池。
6. 并行不降低验证标准：结果须与串行一致（账本 AbsError 等逐位相同，跨进程浮点差 ≤ 1e-12 量级可接受），账本复算、断言、结果审计照常执行。

参考实现：`runM402BCompleteLedgerSensitivity.m`（已在 preflight 4 worker / 6.4 min 验证，与串行 CSV 最大差 5.12e-13）。后续运行器（M4-03B 等）按同一模式推广。

与第 9 节的关系：并行缩短墙钟时间，但不替代长计算交接--若并行后整体仍超 MCP 响应上限，仍按第 9 节由用户在已 attach 的 MATLAB 触发、agent 事后只读核验。

## 11. 建模仿真研究规范

本节是基于本项目全面确认后建立的标准化工作规范，适用于当前及后续所有建模仿真研究活动。本节是项目级唯一的建模仿真研究规范，以 AGENTS.md 第 11 节为权威来源，不在其他文件中重复维护。

### 11.1 文件体系标准

#### 11.1.1 模型文件

- **一种系统结构只有唯一一个 Simulink 模型文件**。各模块尽可能派生于官方案例（`OF-BAT-01`、`OF-DRV-01`、`OF-MCB-01` 等），具备完整的系统结构集成、控制方法和参数匹配能力。
- 模型文件位于 `02_模型/` 目录，`.slxc` 缓存位于同一目录。不允许在根目录或其他目录残留同名 `.slxc` 缓存。
- 不创建并行模型副本以执行不同策略或工况研究；策略和工况变化通过 `SimulationInput` 临时参数注入实现。

#### 11.1.2 脚本文件

- **脚本不承担计算功能，仅辅助 Simulink 模型进行计算**。每个脚本文件承担某一类通用型的辅助功能：
  - 受控运行器（`runM4xxx.m`）：构造 `SimulationInput`、调用 `sim()`/`parsim`、收集结果、写入 CSV、断言验证。
  - 可视化工具（`visualizeM4Results.m`）：只读 CSV 生成图表，不运行仿真。
  - 通用辅助函数（参数表、KPI 计算、窗口分析等）：作为受控运行器的内部子函数，不单独成文件。
- **最低数量的脚本**：拒绝因多策略、多工况等边界条件不同而导致的文件数量爆炸。多工况通过 `createScenarios()` 表结构在一个运行器内统一表达。
- **拒绝重复使用工况依赖性脚本**：临时研究脚本在建立并使用完后直接归档至 `归档/`，后续不再使用。冻结的历史脚本不得在活动 `03_脚本/` 目录中保留。
- M4 的 `runM401CThermalLedger.m`、`runM402BCompleteLedgerSensitivity.m`、`runM403BControlComparison.m` 和 `plot/` 属于冻结参考基线，不再作为活动开发脚本。
- M5 v2 的 `03_脚本/` 初始为空；后续仅在 v2 唯一模型建立后增加必要的受控运行器与只读绘图工具。
- 绘图脚本集中存放在活动平台的 `03_脚本/plot/`；只读 CSV，不运行仿真，不修改模型。

#### 11.1.3 说明文件

- 除 `AGENTS.md` 和 `README.md` 以外的说明文件，按类型分成**规划设计类**和**实施记录类**两类：
  - **规划设计类**（覆盖式更新）：顶层规划、系统定义、资产矩阵、模型影响清单、审计收口分解、参数分层规则。每次执行阶段性任务后，对涉及的规划设计文件做覆盖式更新，以最新版本为准。
  - **实施记录类**（增量式更新）：结构恢复记录、参数敏感性记录、控制策略比较记录。每次执行阶段性任务后，对涉及的实时记录文件做增量式更新，保留历史执行痕迹。
- 每个说明文件禁止无限制叠加重复性内容。过时信息应明确标记为历史或删除。

### 11.2 工况内容标准

每个工况的计算开始前，必须明确三方面内容：

#### 11.2.1 系统初始值

- 明确哪些状态量在仿真开始时被赋值（如 `DCLink_V0=0 V`、`Ambient=263.15 K`、`Coolant=263.15 K`、`T_batt_initial=263.15 K`）。
- 明确哪些状态量保持默认（如机械初始速度为零）。
- 所有初始值必须有来源身份（`OFFICIAL`、`ENGINEERING_DEFAULT`、`PLATFORM_DEFAULT` 等）。

#### 11.2.2 具体控制要求

- 明确被控量（如 `Id_ref` 幅值、频率、占空比、偏置）。
- 明确控制模式（闭环/开环、`EnClosedLoop` 状态）。
- 明确保护/限幅配置（`LimitId`、`LimitFrequency`、`LimitDuty`、`LimitMode`、`LimitStatus`）。
- 明确机械边界条件（零速锁止、允许微动范围）。
- 所有控制参数以 `SimulationInput.setBlockParameter` 临时覆盖，不改写模型默认值。

#### 11.2.3 计算求解器设置

- 所有计算求解器按**瞬态计算**考虑。
- 所有计算求解器的起始时间为 **0 s**。
- 明确仿真终止时间 `StopTime`（如 `0.04 s`、`0.08 s`、`0.20 s`）。
- 明确评价窗口（如 `[0.04, 0.08] s` 的 40 ms 窗口）。
- **StopTime 输出余量规则**：当模型中使用 MATLAB Function（Stateflow EMChart）的 `persistent` 变量做固定窗口累积（如 RMS、DFT、账本子项）时，latched 输出在窗口完成后可能有一拍或一窗口的滞后。此时 `StopTime` 不能等于评价窗口结束时间，须设为 `Window_Stop + Window_Period`（如评价窗口 `[0.03,0.04]s`、窗口周期 `0.01s`，则 `StopTime=0.05s`），确保末样本反映评价窗口结果。M5-03 阶段已验证此规则的必要性（见 `M5-03_执行记录_v01.md` 第 8.3 节）。
- 明确输出记录方式：`logsout` 信号名称、采样时间（如 `2e-6 s`）。
- 考虑物理量随时间的变化曲线，或计算结束时的最终值。

### 11.8 阶段收口与验证工作流（M5-03 经验固化）

本节基于 M5-03 收口过程中遇到的三个阻塞点固化，适用于后续所有阶段的收口验证。

#### 11.8.1 产物存在性先行核验

接手任何阶段收口时，先验证声称的产物是否真实存在且与记录一致，再开始修改或验证：
- CSV 文件是否存在且非空（`04_仿真结果/` 下）。
- 模型块 ID 是否与执行记录一致（通过 `model_overview`/`model_read` 读回）。
- base/model workspace 参数是否就位（`evalin`/`model_query_params` 读回）。
- logsout 信号数是否与记录匹配。
- 执行记录中的数值是否来自实际 CSV（而非设计阶段预写的假设值）。

#### 11.8.2 单工况 diagnose 先行

全量运行前，先用 `diagnose:scenario_id` 模式跑单个工况，逐样本检查关键信号的时序语义：
- 瞬时标志（如窗口完成、触发）是电平型脉冲还是持续型 -> 断言取 `max`/累计计数还是末样本。
- latch 输出是否有滞后 -> StopTime 是否需要留余量（见 11.2 节 StopTime 输出余量规则）。
- 窗口对齐 -> 模型值与独立复算值是否对应同一时间窗口（打印 distinct values + first occurrence time）。

#### 11.8.3 模型值与独立复算值不一致的排查优先级

当受控运行器的断言因"模型值 vs 独立复算值"差异失败时，按以下优先级排查：
1. **窗口对齐**：两者是否对应同一时间窗口？（打印时间序列，检查 first occurrence time）。
2. **latch 滞后**：模型输出是否有一拍/一窗口延迟？（StopTime 是否需要 +Window_Period）。
3. **标志语义**：断言检查的字段是持续型还是瞬时型？（取 max 还是末样本）。
4. **公式实现**：独立复算公式是否与模型代码一致？（逐行对比 phase、amplitude、weight）。
5. **参数传入**：模型参数是否正确解析？（Stateflow Parameter Scope、CompiledSize）。

#### 11.8.4 执行记录与验证运行同步

- 验证结果必须从实际 CSV 回填，不在运行前预写假设数值。
- 执行记录的"完成"以 CSV 存在且受控运行器断言通过为准，不以模型结构建成为准。
- 阶段收口前，执行记录中的每一个数值都应有对应的 CSV 列或 logsout 信号可追溯。
