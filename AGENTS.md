4. 进入正式建模前，先确认当前工作树状态，识别用户已有改动；不得回滚、覆盖或清理与当前任务无关的文件。

## 3. 建模定位

1. 本项目目标不是用一个零维模型证明量产可行，而是在商用车现有高压架构、电池包连接关系、电驱硬件能力和控制约束下，判断车载低温脉冲加热是否存在值得继续投入的工程可行窗口。
2. 现有 0D/L0.5 MATLAB 模型用于温升量级、能耗、电流压力和方案排序粗筛；不能替代后续 L1 电流可实现性模型、Simulink 动态模型、AMESim 一维物理网络、硬件可实现性验证或实验结论。
3. 高频脉冲电流不能等同于普通牵引正弦电流或直流充放电。必须区分峰值、RMS、平均值、母线合成电流、电池支路电流和热等效量。
4. 4500-27 曾发现电池电压平台与逆变器/母线范围存在不一致风险；后续引用任何 811.44 V、618 V、450-750 V、630-919.8 V 等数字前，必须回到当前参数表和供应商资料重新核对。
5. 当前 4500-27 工程边界优先按“三包并联整体输出 + 双电机电驱桥 + 驻车堵转脉冲”理解；单包独立接入、电机集中作用单支路等方案只能作为理论对照，不能写成实车可行方案。

## 4. 宏观研究路线

按“资料盘点 -> 参数注册表 -> 继承 L0.5 粗筛 -> L1 电流可实现性 -> AMESim/Simulink 系统模型 -> KPI 审计 -> 台架问题清单”推进。

### 阶段 A：资料与参数注册

1. 盘点电池、电机、逆变器、多合一、高压盒、整车控制和热管理资料。
2. 建立参数注册表，至少包含参数名、物理含义、单位、来源、适用车型、适用温度/SOC/频率、可信等级和待确认项。
3. 把公开文献、专利、供应商规格书、会议纪要和模型默认值分开标注；不要把占位值写成已确认参数。

### 阶段 B：L0.5 结果继承与复核

1. 复核 `run_4500_27_pulse_heating_screening.m` 的入口、参数脚本、工况矩阵和输出 KPI。
2. 重点复核方案 B：三包并联整体输出 + 双电机同步堵转脉冲。
3. 复核高频发热阻抗 `R_heat_factor`、电机 RMS 限值、三包均流、同步相关性 `rho` 等敏感性假设。
4. 结论只允许表述为“建议进入下一阶段验证”，不能表述为“工程定型”或“安全可行已证明”。

### 阶段 C：L1 电流可实现性

1. 从理想电流转向“目标电流/PWM -> 电流环 -> 电压饱和 -> 最小脉宽/死区/采样 -> 实际电流”的链条。
2. 必须先确认 MCU/逆变器是否允许驻车零速高压使能和高频电流闭环；若直接禁止，应停止复杂模型扩展，转为不可行原因归档。
3. 输出应包括实际电流 RMS/峰值、母线电压裕度、调制裕度、保护触发边界、双电机同步误差影响和电池有效加热电流。

### 阶段 D：AMESim/Simulink 系统模型

1. MATLAB/Simulink 表达控制策略、参数扫描、数据处理、优化、系统级自由建模和后处理。
2. AMESim 表达专业一维物理网络、部件连接、组件匹配、电热网络、液冷/热管理接口和工程系统骨架。
3. 不把 AMESim 当成 Simulink 的简单替代，也不把 Simulink 经验函数当成 AMESim 物理网络替代。
4. 新建 AMESim 模型前先建立 architecture、module、connection、parameter、variable 和 study 注册表；自动生成 `.ame` 只能标记为结构骨架或工作副本，不能直接声明为已标定模型。
5. AMESim 变量名、端口、submodel 和 Data Path 必须从当前安装环境或当前模型动态发现，不硬编码历史 demo 名称。

### 阶段 E：KPI 审计与最小验证

1. KPI 至少覆盖：到 0 C 时间、20/30 min 终温、等效 SOC 消耗、电池发热功率、电机/逆变器损耗、电池支路 RMS/峰值电流、母线电流、电压裕度、保护裕度、析锂/回充风险标志。
2. 每轮输出必须区分已验证事实、基于证据的推断、待确认假设。
3. 无法验证时明确标记未验证原因和剩余风险，不把脚本无报错等同于模型正确。

## 5. MATLAB/Simulink 工作流

1. MATLAB/Simulink 任务先识别入口脚本、目标模型/子系统、关键参数、信号接口、仿真工况、验收指标、允许修改范围和禁止修改范围。
2. 当前 4500-27 粗筛主入口优先按 `零维脉冲加热仿真模型-4500-27型整车/模型目录说明.md` 执行；3806 相关模型优先按其目录说明执行。
3. 读取 `.slx`/`.mdl` 优先使用 `model_overview -> model_read(depth=0/1) -> model_query_params/model_resolve_params`；不默认全模型深读。
4. 结构或参数修改优先使用 SATK MCP 工具、MATLAB/Simulink API 或受控脚本，不直接按 XML/文本修改 `.slx` 或 `.mdl`。
5. 结构性修改后优先运行 `model_check`，再做 read-back verification。
6. 行为验证优先运行最小必要工况、已有 MATLAB 脚本、`sim()`、`model_test` 或 MATLAB 单元测试。

## 6. AMESim 工作流

1. AMESim 任务先固定研究问题、系统边界、保真度、输入输出、控制量、状态变量、参数来源、验证方式和工程决策目标。
2. 已有 `.ame` 模型默认走“GUI 准备工作副本 -> AMEPython 定向读取/赋值/运行 -> KPI 审计 -> 参数恢复或记录”的闭环。
3. 新建模型先形成注册表和接口契约，再用官方 Python/API 或 Circuit API 创建结构骨架。
4. 不按 GUI 点击自动化；不直接改 `.ame` 内部二进制或私有结构。
5. 任何自动生成的 `.ame`、`.cir`、`.sim`、Data Path 输出都必须说明来源、用途和成熟度，不得混入正式结论。

## 7. Codex 审核触发点

OpenCode 推进过程中，遇到以下节点建议请求 Codex 审核：

1. 建立或修改参数注册表后。
2. 首次复现 L0.5 结果后。
3. 准备把 L0.5 假设迁移到 L1/AMESim/Simulink 前。
4. 引入新的电池安全、析锂、BMS 限流、电机退磁、逆变器结温或高压架构判断前。
5. 生成第一版 AMESim/Simulink 结构骨架后。
6. 准备对外输出阶段结论、PPT、会议材料或技术路线建议前。

审核重点是物理语义、证据链、边界条件、参数来源、风险表达和是否过度声称。

1. 不默认导出图片、CSV、报告、模型副本或大批量扫描结果。
2. 不默认回传完整仿真日志、完整 timeseries、完整模型树或工作区 dump。
3. 必须生成中间产物时，放入任务专用输出目录，并在任务结束时说明保留文件及用途。
4. 对 `.docx`、`.xlsx`、`.pptx`、`.pdf`、`.slx`、`.mph`、`.ame` 等结构化文件，使用官方 API、专用工具或受控脚本处理，不按纯文本硬改。
5. Windows 中文路径操作优先使用 `-LiteralPath`、`Join-Path`、`Resolve-Path -LiteralPath`；写入中文内容后必须 read-back verification。
6. 对关键模型、参数表和报告做副本或另存版本前，应说明目的；未经明确授权不自动保存 `.slx`、`.mph`、`.ame` 正式文件。
7. `建模依据资料/`、参数来源清单、专利和论文属于建模依据链，清理工作区时默认保留；优先清理可再生成仿真结果、临时输出、工具缓存、历史汇报和过期归档。
