## 0. 阶段目标

接入 Jian 2023 的负极电位（NEPCS）、端电压（TVCS）和边界频率组合逻辑，建立 UNKNOWN/CANDIDATE/EVALUABLE 三状态安全候选路径，实现拒止逻辑（参数不合格时保持 DeviceProtection_Evaluable=0）。

## 1. 技术方案

### 1.1 设计决策（用户确认）

| 决策项 | 选择 | 理由 |
|---|---|---|
| 核心矛盾处理 | 文献 D1/D2/D3 比值分解 + NaN 传播 | NEPCS 需要负极阻抗，全电池 ECM 无法直接分离；用文献比例建立 CANDIDATE 路径，无三电极数据时 NaN 传播 |
| 实现位置 | blk_844 Battery_Envelope_Screening 内 | 与 M5-03 的 blk_843 并列，不修改物理路径 |
| M4 遗留 blk_712 | 保留为历史对照 | 新增 Compute_Protection_Status 覆盖语义 |
| EVALUABLE 门控测试 | 4.1-E 资格注入工况 | 用 SimulationInput 临时注入有效参数验证门控逻辑可工作 |

### 1.2 NEPCS 候选（Jian 2023 式9/10/11）

```
Re_Z_neg_dc = Re_Z_dc * D3       (文献比例分解)
U_neg_est = |I_ac_amp| * Re_Z_neg_dc   (式9)
I_limit_li_plating = Ue / Re_Z_neg_dc   (式11)
```
Ue=NaN 时 U_neg_est 和 I_limit_li_plating 传播为 NaN。

### 1.3 TVCS 候选（Jian 2023 式6/20）

```
UOCV_eff = UOCV (或 V_batt 代理当 UOCV=NaN)
U_terminal_peak = UOCV_eff + Re_Z_dc * |I_ac_amp|
U_terminal_valley = UOCV_eff - Re_Z_dc * |I_ac_amp|
I_limit_voltage = min((UC-UOCV_eff)/Re_Z_dc, (UOCV_eff-UD)/Re_Z_dc)
```

### 1.4 边界频率候选（Jian 2023 式21）

在 f_eval 频率点中搜索 NEPCS 限流等于 TVCS 限流的交叉点。Ue 或 UOCV 为 NaN 时输出 NaN。

### 1.5 三状态机 + 拒止逻辑

| 状态 | 条件 | DeviceProtection_Evaluable |
|---|---|---|
| UNKNOWN (0) | 核心数学输出 NaN 或混合资格问题 | 0 |
| CANDIDATE (1) | 数学路径已运行，仅 chemistry/waveform 未匹配 | 0 |
| EVALUABLE (2) | 全部资格门控通过 | 1 |

ProtectionReason 为 bitmask：1=U_neg NaN, 2=I_limit_li NaN, 4=f_boundary NaN, 8=chemistry 未匹配, 16=waveform 未适用, 32=RMS 窗口未完成, 64=I_limit_voltage NaN。

RMS_Window_Complete 是瞬时脉冲标志，用 persistent 变量做 latch（一旦完成即保持）。

## 6. 已知事项

1. **D1/D2/D3 为文献中值**：Jian 2023 的 NCM111/石墨单电池数据，不代表目标电芯。CANDIDATE 路径的 U_neg_est 依赖文献比例，不是设备级安全结论。
2. **UOCV=NaN 时 TVCS 用 V_batt 代理**：导致 I_limit_voltage 为负值（V_batt >> UC）。这是无设备数据时的候选路径边界行为，非物理限流值。
3. **f_boundary 在 4.1-E 中为 1Hz**：因 f_eval 第一个点（1Hz）即满足 NEPCS 限流 <= TVCS 限流。这是简化搜索的结果，实际边界频率需要更细的频率分辨率。
4. **RMS_Window_Complete latch**：Protection_Status 块用 persistent 变量做 latch，确保窗口一旦完成即保持。这与 M5-03 的 RMS_WindowComplete_Any 逻辑一致。
5. **波形适用性未验证**：waveform_applicable=0 是 PLACEHOLDER。Jian 2023 的判据基于正弦 AC，v2 的 dSVPWM 产生 PWM 波形，波形等效性需单独验证后才可置 1。

## 7. 结论边界

1. v2 模型已实现安全判据候选路径（NEPCS/TVCS/边界频率）和三状态拒止逻辑（UNKNOWN/CANDIDATE/EVALUABLE），6 工况验证全部通过（CSV 已写入 `04_仿真结果/M5-04_safety_candidate_ledger_v01.csv`）。这是 Reference/Platform 安全候选证据，不是设备析锂安全结论。
2. 默认工况（4.1-A~D/F）中 Ue=NaN，全部安全候选正确传播 NaN，DeviceProtection_Evaluable=0 稳定保持。4.1-E 资格注入测试验证了 EVALUABLE 门控逻辑可工作，但不形成设备安全结论。
3. Jian 2023 的 D1/D2/D3 比值和 UC/UD 截止电压为 LITERATURE 来源（NCM111/石墨、正弦 AC、单电池），不作为通用频率或限流默认值。
4. 未实现控制对照和拓扑替代--属于 M5-05。

## 8. 阻塞点排查与经验积累

本节完整记录 M5-04 执行过程中遇到的全部 8 个阻塞点/错误，按发生顺序排列。其中多个阻塞点源于对 MATLAB/Simulink API 行为的不熟悉，且部分是 M5-03 已遇到问题的复现。这些经验应固化到后续 M5-05 及类似阶段的工作流中。

### 8.1 阻塞点 1：strcat 拼接丢失 char(10) 换行符

**现象**：用 `strcat('line1', char(10), 'line2')` 拼接 MATLAB Function 代码字符串时，`char(10)` 换行符被 strcat 忽略，导致整个函数体变成一行。Stateflow EMChart 的 `Script` 属性接收了无换行的代码，虽然能编译，但可读性为零且难以维护。

**排查**：通过 `rt.find('-isa','Stateflow.EMChart','Path',bp)` 获取 chart 后，检查 `c1.Script` 的 `contains(c1.Script, char(10))` 返回 0，确认换行符丢失。

**解决**：改用文件脚本方式。将代码写入临时 `.m` 文件，用 `char(10)` 变量 `nl` 和方括号拼接 `[... nl ...]`，通过 `run_matlab_file` 执行。文件脚本中 `char(10)` 不会被 strcat 吃掉。

**根因**：MATLAB 的 `strcat` 函数会自动去除尾部空白字符（包括 `char(10)` 换行符）。这是 MATLAB 文档中有记载但容易忽略的行为。

**经验教训**：
- **不要用 strcat 拼接需要保留换行符的字符串。** 应使用 `strjoin(lines, newline)` 或在文件脚本中用 `[... char(10) ...]` 方括号拼接。
- 设置 Stateflow EMChart 代码时，优先用文件脚本方式（`write` + `run_matlab_file`），而非 `evaluate_matlab_code` 内联拼接。
- 设置后应读回验证 `contains(chart.Script, char(10))` 为 true。

### 8.2 阻塞点 2：strcat 中 `(10)` 笔误导致代码语法错误

**现象**：在阻塞点 1 的修复过程中，手动拼接 Protection_Status 代码时，将 `char(10)` 误写为 `(10)`（即 `'    reason = bitor(reason, 16);',(10),`），导致 MATLAB 报语法错误"不支持使用 '=' 运算符"。

**排查**：错误信息直接指向拼接行，检查发现 `(10)` 是纯数字而非 `char(10)`。

**解决**：改用文件脚本方式统一设置所有 4 个 chart 的代码（见阻塞点 1 的解决方案）。

**根因**：手动拼接长字符串时容易出错，且 `evaluate_matlab_code` 内联执行时不利于代码审查。

**经验教训**：
- **多个 MATLAB Function 块的代码设置应统一用文件脚本方式，一次性完成。** 避免在 `evaluate_matlab_code` 中手动拼接长字符串。
- 文件脚本方式允许用 MATLAB 编辑器语法检查预先发现问题。

### 8.3 阻塞点 3：From_I_batt_safety (blk_962) 悬空输出端口

**现象**：blk_844 model_check 报错 `unconnected_output: Port 'y1' of block 'From_I_batt_safety' is not connected`。

**排查**：回顾设计，I_ac_amp 输入用的是 I_batt_rms（窗口 RMS 值，blk_969 From_I_batt_rms_safety），而非瞬时 I_batt（blk_962 From_I_batt_safety）。blk_962 是设计阶段多余创建的块。

**解决**：删除 blk_962。

**根因**：在规划信号路由时，同时创建了 I_batt（瞬时）和 I_batt_rms（窗口）两个 From 块，但实际只有 I_batt_rms 被使用。未在创建前确认每个 From 块的消费端。

**经验教训**：
- **创建 From 块前应先确认其消费端（哪个 MATLAB Function 块的哪个输入端口）。** 避免创建无消费端的悬空块。
- model_check 的 `unconnected_ports` 检查能有效捕获此类问题，应在每次结构修改后立即执行。

### 8.4 阻塞点 4：信号记录配置方式探索耗时

**现象**：M5-03 已有 36 个 logsout 信号，但花费大量时间探索它们是如何配置的。尝试了以下多种方式均未找到：
- SignalSpecification 块的 DataLogging 参数（不存在）
- 信号线的 DataLogging/SignalName 属性（line 句柄无此参数）
- base/model workspace 中的 Simulink.Signal 对象（不存在）
- ToWorkspace 块（不存在）
- root Outport（不存在）
- Goto 块的信号属性（无 DataLogging）
- ModelLoggingInfo（返回 0 个信号）
- `get(lh)` 获取 line 属性（被 MATLAB 拦截）

**排查**：最终通过遍历所有块的 Outport 端口，检查 `get_param(portHandle, 'DataLogging')` 属性，发现 M5-03 的 36 个信号全部通过**端口级** `DataLogging=on` + `DataLoggingNameMode=Custom` + `DataLoggingName=<name>` 配置。

**解决**：对 9 个新 SigSpec 块的 Outport 端口执行同样的配置：
```matlab
ph = get_param(h, 'PortHandles');
outPh = ph.Outport(1);
set_param(outPh, 'DataLogging', 'on');
set_param(outPh, 'DataLoggingNameMode', 'Custom');
set_param(outPh, 'DataLoggingName', 'U_neg_est_V4K');
```

**根因**：M5-01 迁移时继承了 M4 的信号记录配置方式，但未在执行记录中明确记录"如何配置新信号的 DataLogging"。后续阶段接手时需要重新探索。

**经验教训**：
- **v2 模型的信号记录通过端口级 DataLogging 配置，不是 SignalSpecification、Simulink.Signal 或 line 属性。** 配置方式为：`set_param(portHandle, 'DataLogging', 'on')` + `DataLoggingNameMode='Custom'` + `DataLoggingName='<signal_name>'`。
- **此配置方式应记录到接口契约或研究规范中，避免后续阶段重复探索。** 建议在 AGENTS.md 第 11 节或接口契约中补充"信号记录配置标准"。
- 探索信号记录配置时，最快的方式是遍历所有块的 Outport 端口检查 `DataLogging` 属性，而非尝试 line、SigSpec 或 workspace 对象。

### 8.5 阻塞点 5：table 列索引在单行时返回非标量

**现象**：`verifyResults` 中 `results.ProtectionStatus(idxA)` 在单行 table（diagnose 模式）时返回 1x1 table 而非标量 double，导致 `assert(protStatus(idxA) == 0)` 报错"条件输入参数必须可转换为标量逻辑值"。

**排查**：通过手动构建单行 table 测试，发现 `table.Col(logical_scalar)` 的返回类型取决于 table 的行数和列的数据类型。即使列是 double，单行 table 的逻辑索引也可能返回 table 子集而非标量。`table2array` 可以强制转换为数组。

**解决**：用 `table2array(results(:, {'ProtectionStatus'}))` 提取为纯 double 数组，用 `find(scenId == "4.1-X", 1)` 获取标量索引。

**根因**：MATLAB table 的列索引行为与 struct array 和普通数组不同。`table.Col(logical)` 返回子 table，而 `table.Col{logical}` 或 `table2array` 才返回值。

**经验教训**：
- **在 verifyResults 中操作 table 列时，应先用 `table2array` 提取为纯数组。** 避免直接用逻辑索引访问 table 列。
- 用 `find(condition, 1)` 获取标量索引，而非直接用 logical 向量索引。
- M5-03 的运行器未遇到此问题，因为 M5-03 的 `verifyResults` 使用 `all(results.Col == value)` 形式（all 接受向量），而 M5-04 的断言需要逐工况检查标量值。

### 8.6 阻塞点 6：verifyResults 在 diagnose 模式检查不存在的工况

**现象**：diagnose 模式只运行一个工况（如 4.1-B），但 `verifyResults` 检查所有 6 个工况的断言（包括 4.1-A），导致不存在的工况断言失败（`idxA` 为空，`protStatus(idxA)` 报错）。

**排查**：错误信息"4.1-A did not produce UNKNOWN protection status"在运行 4.1-B diagnose 时出现，说明 verifyResults 在检查 4.1-A 但 results 中只有 4.1-B。

**解决**：所有工况特定断言改为条件性检查：`idxA = find(scenId == "4.1-A", 1); if ~isempty(idxA), assert(...), end`。

**根因**：verifyResults 设计为全量验证，但 diagnose 模式只运行单个工况。两者不兼容。

**经验教训**：
- **verifyResults 中的工况特定断言必须用 `if ~isempty(idx)` 条件保护，使其在 diagnose 模式下只验证实际存在的工况。**
- M5-03 的运行器未遇到此问题，因为 M5-03 的断言全部用 `all(...)` 形式（对空数组返回 true），而 M5-04 的断言需要逐工况检查标量值。
- diagnose 模式的设计目的是快速验证单个工况的信号时序语义，verifyResults 应支持此模式。

### 8.7 阻塞点 7：RMS_Window_Complete 瞬时标志（M5-03 阻塞点 2 复现）

**现象**：4.1-E（资格注入测试）中 ProtectionReason=32（bit 5: RMS 窗口未完成），但 `RMS_WindowComplete_Any=1` 表明窗口实际已完成。Protection_Status 块直接用末样本的 RMS_Window_Complete 信号（瞬时脉冲标志），在窗口边界外为 0。

**排查**：这与 M5-03 阻塞点 2（`M5-03_执行记录_v01.md` 第 8.2 节）完全相同。RMS_Window_Complete 是电平型脉冲，仅在 10ms 窗口边界处（t=0.01/0.02/0.03s）为 1，其余时间为 0。Protection_Status 块作为 MATLAB Function 逐样本计算，末样本（t=0.05s）的 RMS_Window_Complete 为 0。

**解决**：在 Protection_Status 块中用 `persistent` 变量做 latch：
```matlab
persistent wc_latched;
if isempty(wc_latched), wc_latched = 0; end
if RMS_Window_Complete > 0.5, wc_latched = 1; end
% 后续用 wc_latched 替代 RMS_Window_Complete 做门控判断
```

**根因**：M5-03 已在运行器侧用 `max(wcFlag) > 0.5` 解决了此问题（`RMS_WindowComplete_Any`），但 M5-04 新增的 Protection_Status 块是模型内的 MATLAB Function，无法用 `max` 逻辑，必须用 persistent latch。

**经验教训**：
- **这是 M5-03 阻塞点 2 的第二次复现。** 瞬时脉冲标志不能直接用作门控条件，必须在模型内用 persistent latch 或在运行器侧用 max 逻辑。
- **新增 MATLAB Function 块如果依赖瞬时标志（如 RMS_Window_Complete、窗口完成触发、边沿检测等），应在首次编写代码时就预先考虑 latch 机制。** 不要等到验证失败才发现。
- 建议在 AGENTS.md 第 11.8 节"阶段收口与验证工作流"中补充：新增 MATLAB Function 块依赖瞬时标志时，必须在代码中用 persistent latch。

### 8.8 阻塞点 8：NaN ~= -1 返回 false 的 MATLAB 语义陷阱

**现象**：运行器中用 `scenario.Ue_override ~= -1` 判断是否需要通过 `setVariable` 注入 Ue 覆盖值。4.1-A~D 和 4.1-F 的 `Ue_override = NaN`，但 `NaN ~= -1` 在 MATLAB 中返回 `false`（logical 0），导致覆盖未执行。

**排查**：实际验证中 4.1-A~D 的 U_neg_est 正确为 NaN（因为 base workspace 的 `Ue_neg_default` 默认就是 NaN），所以此问题未导致错误结果。但如果 base workspace 默认值不是 NaN，此 bug 会导致覆盖未执行。

**解决**：此问题实际未造成错误（因为默认值与覆盖值一致），但应在代码中用 `~isempty(scenario.Ue_override) && ~(isnan(scenario.Ue_override) && scenario.Ue_override ~= -1)` 或更清晰的哨兵值（如用 `-999` 表示"不覆盖"而非 `-1`）。

**根因**：MATLAB 中 `NaN` 与任何数的比较（包括 `~=`）都返回 `false`。这是 IEEE 754 标准行为，但容易被忽略。

**经验教训**：
- **MATLAB 中 `NaN ~= x` 永远返回 false。** 判断"是否需要覆盖"时，如果覆盖值可能为 NaN，不能用 `~=` 比较。
- 运行器的哨兵值应选择不可能出现在实际数据中的值（如 `-999` 或 `[]`），而非 `-1`（因为 `-1` 可能是有效的物理值）。
- 此问题未导致结果错误（因默认值与覆盖值一致），但属于潜在的逻辑缺陷。

### 8.9 工作流强化建议（适用于后续 M5-05 及类似阶段）

基于上述 8 个阻塞点，提出以下工作流强化建议：

1. **MATLAB Function 代码设置标准**：
   - 多个 MATLAB Function 块的代码设置统一用文件脚本方式（`write` + `run_matlab_file`），不用 `evaluate_matlab_code` 内联拼接。
   - 文件脚本中用 `nl = char(10)` 和方括号拼接 `[... nl ...]`，不用 `strcat`。
   - 设置后读回验证 `contains(chart.Script, char(10))` 为 true。

2. **信号记录配置标准**：
   - v2 模型的信号记录通过端口级 `DataLogging=on` + `DataLoggingNameMode=Custom` + `DataLoggingName=<name>` 配置。
   - 新增 logsout 信号时，对 SigSpec 块的 Outport 端口执行此配置。
   - 建议将此配置方式记录到 AGENTS.md 第 11 节或接口契约中。

3. **From/Constant 块创建前确认消费端**：
   - 创建 From/Constant 块前，先确认其消费端（哪个 MATLAB Function 块的哪个输入端口）。
   - 避免创建无消费端的悬空块，model_check 后立即检查 unconnected_ports。

4. **verifyResults 支持 diagnose 模式**：
   - 工况特定断言用 `if ~isempty(idx)` 条件保护。
   - 用 `table2array` 提取列为纯数组，用 `find(condition, 1)` 获取标量索引。
   - 避免直接用 logical 向量索引 table 列。

5. **瞬时标志 latch 机制**：
   - 新增 MATLAB Function 块如果依赖瞬时脉冲标志（如 RMS_Window_Complete），必须在代码中用 persistent latch。
   - 不要等到验证失败才发现此问题。
   - 此规则应补充到 AGENTS.md 第 11.8 节。

6. **运行器哨兵值选择**：
   - 用 `-999` 或 `[]` 表示"不覆盖"，不用 `-1`（可能是有效物理值）。
   - 注意 `NaN ~= x` 永远返回 false 的 MATLAB 语义。
