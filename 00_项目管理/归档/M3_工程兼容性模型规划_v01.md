# M3 工程兼容性模型规划 v01

日期：2026-07-15
状态：既有规划基线，阶段 2 已确认，待阶段 4 顶层规划冻结后重审；不构成当前实施授权

## 1. 阶段定义

本文件是既有 **M3 工程兼容性模型规划基线**。当前项目阶段为阶段 4 共同对齐顶层建模规划。目标不是立即建立具体产品虚拟样机，也不是继续修补 V4-I，而是先定义一个可替换、可审计、可验证的通用低温脉冲加热系统模型框架。

`车载脉冲加热技术simlink模型/02_模型/pulse_heating_official_spine_v04.slx` 是当前唯一活动的 V4 受控工程基线，后续工程化改进以它为核心直接迭代。`归档/M2_V4-I/` 只保存冻结证据、历史脚本、损坏原件和恢复副本；不能把活动 V4 与归档证据混为一类。

本文件保留此前 G0-G3、接口、参数和验证分析作为规划基线，但当前阶段不把其中的“已完成”“首轮实现顺序固定”或脚本/Precheck 状态解释为已确认路线。阶段顺序必须遵循：分析清理 -> 工程化定义草案 -> 资产归档收口 -> 共同对齐的顶层建模规划。阶段 4 完成前不得修改 V4、创建新脚本或运行仿真。

## 2. M3 正式定义

车载低温脉冲加热模块化集成系统工程模型：在明确低温驻车脉冲加热机理、系统边界、运行工况和设计目标的前提下，将动力电池、高压配电与 DC-link、逆变器、PMSM、控制器、热管理、保护逻辑及必要的简化机械约束进行多域耦合；以统一的参数来源、接口契约、约束包线和验证规则，评价系统可实现性、参数匹配性、能量与热行为、控制稳定性及工程风险。

M3 服务于部件选型、参数分配、方案比较和台架验证准备，不直接输出具体产品定型、析锂安全、寿命或量产许可结论。

## 3. 已确认目标与非目标

### 3.1 已确认目标

1. 建立通用、模块化的工程兼容性模型架构，而非绑定某一车型或历史产品参数。
2. 建立电池、DC-link、逆变器、电机、控制器、热管理和保护之间的接口契约与共同工况包线。
3. 支持受控替换电池、电机、逆变器和控制器的参数配置，不将 M2 冲突值升级为平台默认值。
4. 使 RMS、能量积分、损耗、温度、控制时序和保护动作具有可审计语义。
5. 为后续实测数据拟合、台架验证和具体产品适配保留明确入口。

### 3.2 非目标

1. 不在当前阶段复刻完整机械几何、壳体、支架、线束三维布局或有限元结构强度。
2. 不将历史 4500-27、L0.5 或公司资料作为默认架构和默认参数。
3. 不默认承诺 M3 必须沿用 M2 的三条官方案例组合、频率、拓扑或参数值。
4. 不在缺少真实数据时声称产品级安全、寿命、NVH、退磁或控制量产可行。

## 4. 系统边界与目标结构

M3 的最低系统边界由以下模块构成：

| 模块 | 最低工程表达 | 主要接口 |
|---|---|---|
| 需求与配置 | 工况、目标、配置 ID、参数来源和适用范围 | 目标温升、环境温度、SOC、模式、验证配置 |
| 运行与安全监督 | 使能、限流、降额、停机、故障优先级 | BMS/VCU 约束、状态、保护动作 |
| 电池与热管理 | 电池等效电路、热网络、冷却与 BMS 能力包线 | V/I/SOC/T、冷却条件、允许电流 |
| 高压配电与 DC-link | 母排/线束/接触器等效、传感、储能和损耗 | 电压、电流、功率、压降、开断状态 |
| 逆变器、电机与简化机械边界 | 调制、器件损耗、PMSM、驻车零速/锁止和热边界 | 门极或调制、相电流、转速、转矩、温度 |
| 控制与采样 | 电流指令、PI、PWM、采样、死区、最小脉宽和饱和 | 参考值、反馈、采样周期、调制裕度 |
| 验证与审计 | KPI、统一时间基准的能量积分、收敛与配置记录 | 峰值/平均/RMS、能量、残差、误差、风险 |

完整机械集成暂不建模为三维几何，但其影响必须通过电阻、电感、热容、热阻、冷却边界、惯量、锁止条件和允许持续时间等参数化边界进入系统模型。

## 6. M2 冻结基线的使用边界

M2/V4-I 只提供已发现问题和可复用的诊断经验，不提供 M3 默认参数或已验证设计结论。

| M2 发现 | M3 中的角色 |
|---|---|
| 电池容量、限流基值、拓扑与损耗系数曾存在相互矛盾的历史残留 | 参数身份和限流链必须在 M3 配置基线中统一；不得由该残留反推当前电池拓扑或能力 |

| 电压等级、控制硬件身份与时间参数曾存在冲突 | M3 必须建立跨设备电压、电流、热和时序兼容性矩阵；不得由官方示例或历史记录推断当前平台角色 |
| 跟踪误差、保护瞬态、结构 warning 和求解器未验收 | M3 必须定义相应的验证门槛，不能直接继承为通过 |

## 7. 实施门槛

在开始任何 M3 `.slx` 新建、复制或结构修改前，必须依次完成：

1. **G0 系统定义**：明确工程问题、默认架构、工况包线、模型边界、非目标和结论等级。
2. **G1 接口契约**：定义所有关键电气、控制、热、机械和保护接口的物理语义与约束。
3. **G2 参数配置基线**：确定平台默认配置的来源规则、拓扑、设备身份、占位策略和替换规则。
4. **G3 验证计划**：定义结构检查、RMS、能量、控制、热、收敛和数据验证的通过条件。

完成 G0-G3 后，才允许决定使用新 M3 模型、受控复制的工作副本，或经批准复用 M2 的局部结构。

## 8. G0 系统定义草案

状态：既有 G0 草案基线，待阶段 2 和阶段 4 共同复核；当前不授权 `.slx` 实现。

### 8.1 工程问题

M3 要回答的不是“给定理想电流能否升温”，而是：

> 对一个给定的电池包-电驱模块配置，在低温驻车条件下，脉冲加热命令能否通过实际的高压能量路径、逆变器、电机和控制链形成；其电压、电流、功率、热、时序和保护约束中，哪些约束先成为限制因素；哪些部件参数或接口需要调整、验证或替换。

M3 的第一轮输出是兼容性判断、约束主导项、风险和数据缺口，不是最优控制参数或产品定型值。

### 8.2 最小可验证单元

M3 默认采用 **单电池包-单电驱模块** 作为最小可验证单元：

```text
运行与安全监督
       |
电池与热管理 <-> 高压配电/DC-link <-> 逆变器 <-> PMSM + 驻车机械边界
       |                    |                  |
       +---------------- 控制、采样与保护 ----------------+
```

该单元不是特定产品拓扑。它应保留用于未来扩展的模块级接口，使多电池包、多电驱支路或共享高压母线能够通过配置组合和上层连接扩展，而不是复制或改变单元内部物理语义。

### 8.3 模型边界

| 范围 | M3 表达 | 外部边界或不表达内容 |
|---|---|---|
| 电池 | 电压、支路电流、SOC、等效电路损耗、热状态、冷却边界与允许能力 | 具体电芯的电化学析锂机理、老化和认证结论 |
| 高压路径 | 母排/线束/接触器等效、DC-link 储能、传感与损耗 | 三维线束布局、EMC 和接触器详细电弧 |
| 电驱 | 逆变器调制与损耗、PMSM 电磁与热、零速或锁止机械边界 | 完整传动系统、壳体结构、NVH 与机械有限元 |
| 控制与保护 | 命令、采样、限幅、PWM、保护优先级、故障退出 | 量产 MCU 代码、通信栈和诊断覆盖率认证 |
| 热管理 | 电池、电驱的热源和参数化热边界 | 冷却板流道 CFD 和完整车身热环境 |
| 验证 | 接口、能量、RMS、收敛、参数和数据证据审计 | 无实测支持的产品安全或寿命放行 |

### 8.4 外部输入、输出与状态

| 分类 | 最低内容 | M3 角色 |
|---|---|---|
| 配置输入 | 电池、电驱、控制、热管理和高压路径的配置 ID | 选择受控参数组合，不直接散布常数 |
| 工况输入 | 环境温度、初始 SOC、冷却条件、驻车/锁止状态、脉冲加热请求 | 定义可复现的运行场景 |
| 约束输入 | BMS 可用电流/电压/SOC/温度包线、逆变器与电机限制、控制使能条件 | 必须显式声明来源或占位状态 |
| 控制输入 | 模式请求与候选脉冲目标；具体 d/q 策略和频率暂不预设 | 进入控制链前接受监督与限幅 |
| 测量输出 | 电池端、DC-link、逆变器、电机和热管理的 V/I/P/T/SOC/状态 | 用于控制反馈和接口验证 |
| 审计输出 | 峰值、平均值、窗口 RMS、能量积分、损耗、温升、约束裕度、保护事件和数据缺口 | 形成工程兼容性结论 |

### 8.5 运行模式与首轮工况类别

| 模式 | 目的 | 当前要求 |
|---|---|---|
| Idle | 无加热请求或禁止运行 | 不产生加热命令，定义静态基线 |
| Precheck | 校验配置、边界和使能 | 任一关键约束未知或不满足时禁止进入 Active |
| Active | 执行受监督的脉冲加热候选命令 | 记录实际电流、能量、损耗和所有限幅 |
| Derate | 某一约束接近边界 | 明确降额原因、优先级和退出条件 |
| Stop/Fault | 保护退出或外部禁止 | 明确安全状态和允许的重新使能条件 |

首轮工况只分类，不在 G0 预设历史频率、容量、电压或设备额定值：

1. 静态和零命令基线。
2. 低温启动的标称加热候选工况。
3. 电池、母线、逆变器、电机、控制时序和热边界分别主导的限制工况。
4. 约束缺失、禁止使能、传感异常或保护退出工况。
5. 参数替换与不确定性工况，用于识别对结论最敏感的接口。

### 8.6 结论等级

| 等级 | 可表述内容 | 不可表述内容 |
|---|---|---|
| C0 结构事实 | 模块、端口、配置和信号是否存在 | 物理行为或工程可行性 |
| C1 数值与接口事实 | 仿真是否到达终点、量纲和数值是否可审计 | 参数正确性、设备匹配或安全 |
| C2 工程兼容性推断 | 在明确假设和工况下的约束主导项、风险和可替换性 | 未验证设备的产品定型或安全许可 |
| C3 数据验证结论 | 与已定义的台架、供应商或实测数据比较后的适用范围和误差 | 超出数据范围的外推结论 |

M3 在未取得独立数据前，最高只输出 C2 结论。

### 8.7 G0 通过条件

G0 通过需确认：

1. 单电池包-单电驱模块是 M3 的最小可验证单元，多模块仅通过接口和配置扩展。
2. M3 以驻车低温脉冲加热的工程兼容性判断为首轮用途。
3. 电池、高压路径、电驱、控制、热管理和保护的系统边界与外部边界如本节所述。
4. 首轮不预设历史车型、具体产品、固定频率、容量、电压、拓扑或控制 d/q 策略。
5. 在 G1-G3 完成前，不创建或修改 M3 `.slx`。

### 8.8 已确认的首轮工程路线

| 决策域 | 已确认路线 | 对 M3 的约束 |
|---|---|---|
| 首轮决策目标 | 兼容性优先 | 首要输出是电压、电流、功率、热、时序和保护的约束主导项与裕度；加热收益仅作为受约束的审计输出，不进行产品性能寻优 |
| 候选电流路径 | 接口化、可替换的候选路径 | 不预设 d/q 轴、相电流、外部功率路径、频率或波形；每种路径必须由 ControlConfig 声明其参考语义、执行链和适用边界 |
| 电驱保真度 | 平均值主线，PWM 代表工况抽查 | 主线采用含 DC-link 动态、离散控制和 PMSM/热边界的平均值电驱表达；PWM/开关级只用于已通过平均值 Precheck 的代表工况，以复核时序、纹波、死区或损耗假设 |
| 证据路线 | 结构/文献先行，独立台架验证后升阶 | 官方资产只提供结构、接口或实现参考；文献只提供机理和适用条件内的量级；没有独立数据时最高为 C2，C3 必须等待可追溯台架、供应商或实测数据 |

“平均值主线”不允许省略采样、限幅、保护、DC-link 储能或损耗口径；“PWM 抽查”也不能将单一高频仿真结果提升为设备额定、安全或量产结论。

## 9. G1 接口契约

状态：既有 G1 草案基线，待阶段 2 工程化定义共同复核；当前不冻结后续模型接口实现。

### 9.1 通用约定

1. 跨模块交换的连续物理量默认使用 SI 单位：电压 V、电流 A、功率 W、能量 J、温度 K、热流 W、时间 s、频率 Hz、角度 rad、转速 rad/s、转矩 N*m、电阻 Ohm、电感 H、热容 J/K、热阻 K/W。
2. 报告可额外显示 `degC`，但模型内部、接口字段和比较运算必须明确单位，禁止 K 与 degC 混用。
3. SOC 使用无量纲分数 `[0, 1]`；占空比、降额系数和调制指数为无量纲量，并分别注明其定义域。
4. 跨模块接口使用物理量和显式单位，不使用未经定义的 per-unit、归一化值或历史总线字段。
5. 所有配置、测量、命令、状态和审计信号均需有有效性状态；关键输入未知、越界或来源不明时不得默认允许 Active 模式。

### 9.2 电气方向与端口功率口径

| 接口 | 正方向与正功率定义 | 最低输出 |
|---|---|---|
| 电池 -> 高压路径 | `I_batt > 0` 表示电流从电池正端流向高压路径；`P_batt_terminal = V_batt * I_batt > 0` 表示电池端向外输出电功率 | `V_batt`、`I_batt`、`P_batt_terminal`、SOC、温度、能力状态 |
| 高压路径 -> 逆变器 | `I_dc > 0` 表示电流由 DC-link 流入逆变器；`P_dc = V_dc * I_dc > 0` 表示逆变器吸收 DC 电功率 | `V_dc`、`I_dc`、`P_dc`、母线损耗、储能状态 |
| 逆变器 -> PMSM | 相电流正方向由逆变器 AC 端指向电机端；`P_ac > 0` 表示电功率流入电机 | `i_abc`、`i_dq`、`P_ac`、调制状态、器件损耗 |
| PMSM -> 机械边界 | `T_e * omega_m > 0` 表示电机向机械端输出机械功率；驻车锁止时 `omega_m = 0`，反作用转矩不应被误记为机械能输出 | `omega_m`、`T_e`、`P_mech`、锁止状态 |
| 热接口 | `Qdot_gen > 0` 表示部件内部产生热；`Qdot_reject > 0` 表示热量由部件排向冷却或环境边界 | 温度、发热、排热、热边界状态 |

电池内部损耗、DC-link 储能、逆变器损耗、电机损耗、机械储能和热网络储能必须分域记录。`P_batt_terminal` 不能直接代替电池化学能变化，也不能与某一损耗项直接比较后声称系统能量闭合。

### 9.3 电流、功率与能量审计口径

| 指标 | 定义 | 使用限制 |
|---|---|---|
| 峰值 `x_peak` | 在明确窗口内的 `max(abs(x))` | 必须同时记录窗口和采样来源 |
| 有符号平均 `x_mean` | `1/Tw * integral(x(t) dt)` | 不得代替 RMS 或峰值 |
| 窗口 RMS `x_rms` | `sqrt(1/Tw * integral(x(t)^2 dt))` | 必须记录窗口 `Tw`、起止时刻、采样/积分方法和适用对象 |
| 平均功率 `P_mean` | `1/Tw * integral(P(t) dt)` | 端口功率与损耗功率分开报告 |
| 能量 `E` | `integral(P(t) dt)` | 与对应端口方向、时间窗口和储能变化同时报告 |
| 热等效量 | 由明确热模型或 `integral(I(t)^2 * R(... ) dt)` 推导 | 不得用瞬时电流绝对值或未定义电阻代理替代 |

每次能量审计至少区分：电池端传递能量、高压路径损耗与储能变化、逆变器损耗与 AC 端传递能量、电机电损耗/机械功率/储能变化，以及各热网络的能量变化与排热。只有在同一时间窗口、相同符号约定和已列出的储能项下计算的残差才可称为能量残差。

### 9.4 时间与采样契约

| 时间域 | 角色 | G1 要求 |
|---|---|---|
| 连续物理时间 | 电池、电力电子、电机、热和机械状态演化 | 使用求解器时间基准；记录 solver、容差、最大步长和事件处理方式 |
| PWM 时间基准 | 载波、门极、死区和最小脉宽 | 由配置给出 `T_pwm` 或等价频率；明确与控制更新的关系 |
| 电流控制时间基准 | 采样、滤波、控制器更新、限幅和抗饱和 | 由配置给出 `Ts_ctrl`；每个反馈信号须注明采样、保持和延迟语义 |
| 监督与保护时间基准 | Precheck、限流、降额、停机和重新使能 | 由配置给出 `Ts_sup` 和必要的确认/延时规则；硬保护不能被慢监督逻辑掩盖 |
| 审计与记录时间基准 | RMS、能量和事件记录 | 记录时标必须足以重建窗口积分和保护事件顺序，不能只记录最终样本 |

G1 不固定任何采样频率或求解器。G2 必须为每一配置建立 `Ts_meas`、`Ts_ctrl`、`T_pwm`、`Ts_sup`、记录步长和物理求解器设置，并说明它们的同步或速率转换关系。

### 9.5 模块级接口

| 接口 ID | 生产者 -> 消费者 | 最低字段 | 约束 |
|---|---|---|---|
| I0 配置 | 配置管理 -> 全模块 | 配置 ID、版本、来源状态、有效性、参数集引用 | 无有效配置不得进入 Precheck 之后的模式 |
| I1 工况与请求 | 外部工况 -> 监督层 | 环境温度、初始 SOC、冷却条件、驻车/锁止、`CandidatePath_ID`、候选参考语义和请求 | 所有输入必须带单位和有效性；候选脉冲目标不绕过监督层 |
| I2 监督与限制 | 电池/BMS、电驱、热管理 -> 监督层 | 可用电压/电流/温度/SOC 包线、硬/软限制、故障与传感有效性 | 来源为 `PLACEHOLDER` 的关键保护边界应阻止 Active 或被显式标注为研究假设 |
| I3 有效命令 | 监督层 -> 控制层 | 模式、使能、已核验的 `CandidatePath_ID`、有效参考、降额系数、限幅原因、停机原因 | 控制层只执行有效命令；每个限幅必须可追溯至约束来源 |
| I4 控制与电驱 | 控制层 <-> 逆变器/PMSM | 参考值、参考语义、测量值、采样状态、调制状态、饱和状态、可实现性状态 | 控制策略可替换；每个路径须定义参考坐标/物理含义、反馈极性、延迟、单位和抗饱和行为 |
| I5 电池与高压路径 | 电池 <-> 高压路径 <-> 逆变器 | 各端 `V/I/P`、开断状态、DC-link 储能、压降与损耗 | 使用 9.2 的正方向；任何旁路电源或测试源必须显式标记，不能伪装为电池路径 |
| I6 热接口 | 电池、电驱 <-> 冷却/环境 | 温度、发热、排热、冷却输入、热边界与有效性 | 热模型使用的电流口径、热容和热阻必须与配置身份一致 |
| I7 审计 | 全模块 -> 验证与记录 | 峰值/平均/RMS、能量、损耗、温升、裕度、事件、数据缺口 | 同一指标只保留一个权威定义，禁止历史 KPI 名称复用但语义变化 |

### 9.6 模式、保护与优先级

| 当前模式 | 允许进入条件 | 退出或转移条件 |
|---|---|---|
| Idle | 无有效加热请求，或系统已复位 | 有请求时进入 Precheck |
| Precheck | 配置、输入、测量和关键边界可用 | 全部满足进入 Active；否则保持 Idle 或转 Stop/Fault |
| Active | Precheck 通过且未触及软/硬约束 | 软约束触发转 Derate；硬约束、故障或请求撤销转 Stop/Fault 或 Idle |
| Derate | Active 中触及可恢复软约束 | 约束恢复可返回 Active；继续恶化或硬约束触发转 Stop/Fault |
| Stop/Fault | 硬约束、关键传感/配置无效、外部禁止或故障 | 只有明确复位、重新 Precheck 和允许条件满足后才能重新使能 |

暂定优先级为：配置或关键传感无效/故障停机 > 硬件硬限制 > 明确 BMS/外部安全限制 > 热限制 > 控制可实现性限制 > 性能目标。G2 必须为各配置建立具体阈值、去抖/延时、责任模块和可恢复性；未定义的优先级不得用默认常数补全。

### 9.7 参数与数据有效性契约

1. 电池、DC-link、逆变器、PMSM、控制和热管理配置具有独立 ID、版本和来源标签；组合配置必须记录兼容性检查结果。
2. 关键使能、限流、热和保护参数必须有设备或平台身份、单位、适用范围和验证状态；缺失时不能自动进入 Active。
3. 允许研究性 `PLACEHOLDER` 参数存在，但必须输出数据缺口状态，并将结论限制在相应等级。
4. 参数替换后，至少重新检查相关电压、电流、功率、热、频率、采样、保护、RMS 和能量审计接口。
5. 所有输入、状态、输出和配置均须可追溯到唯一配置 ID；禁止从模型工作区、脚本和模块掩码中隐式叠加不一致的同名常数。

### 9.8 G1 通过条件

G1 通过需确认：

1. 9.1-9.7 的单位、方向、RMS/能量口径、时间域和模式语义可作为后续唯一接口定义。
2. 所有跨模块接口均可归入 I0-I7，且不存在未定义来源、单位或方向的关键物理量。
3. 关键限制在缺失时采取保守禁止或明确研究假设，而非默认允许。
4. G2 将为每一候选配置补齐参数、阈值、时序和设备身份；G1 不以历史数值填补这些空位。
5. G3 将为每项接口定义验证证据和通过条件。

## 10. G2 参数配置基线

状态：既有 G2 草案基线，待阶段 2 工程化定义和阶段 3 资产收口后复核；当前不创建配置实现。

### 10.1 配置对象与组合 ID

M3 配置采用“部件配置独立、系统组合受控”的形式：

```text
SystemConfig_ID
  = BatteryConfig_ID
  + HVPathConfig_ID
  + InverterConfig_ID
  + MotorConfig_ID
  + ControlConfig_ID
  + ThermalConfig_ID
  + ProtectionConfig_ID
  + ScenarioConfig_ID
```

`SystemConfig_ID` 必须唯一记录所有子配置 ID、版本、来源状态、兼容性检查结果和创建日期。任一子配置变化都生成新的系统组合版本；不得在同一 ID 下隐式替换参数。

| 配置对象 | 责任边界 | 最低身份字段 |
|---|---|---|
| BatteryConfig | 电池等效电路、拓扑、能力、SOC 与热边界 | 电池对象/平台身份、拓扑、容量语义、参数来源、适用 T/SOC/频率 |
| HVPathConfig | 接触器、母排/线束、DC-link、传感与高压路径 | 拓扑、额定/允许范围、R/L/C、损耗和测量位置 |
| InverterConfig | 功率器件、DC/AC 侧能力、调制与损耗 | 器件或平台身份、允许 V/I、开关模型层级、损耗与热边界 |
| MotorConfig | PMSM 电磁、热与驻车机械边界 | 电机或平台身份、相数、极对数、R/L/磁链、热与锁止边界 |
| ControlConfig | 候选路径、采样、控制更新、PWM、限幅和抗饱和 | 控制结构版本、`CandidatePath_ID`、参考语义、`Ts_meas`、`Ts_ctrl`、`T_pwm`、反馈与执行语义 |
| ThermalConfig | 环境、冷却、热容、热阻、温度传感 | 热网络范围、边界类型、冷却条件、测点与适用范围 |
| ProtectionConfig | 进入、降额、停机与复位策略 | 保护结构版本、阈值来源、优先级、延时、可恢复性 |
| ScenarioConfig | 初始状态、外部工况和候选请求 | 环境、SOC、驻车状态、冷却状态、请求类型与观察窗口 |

### 10.2 通用参数记录格式

每个参数记录须满足以下字段；缺失字段必须显式写为 `PLACEHOLDER` 或 `UNKNOWN`，不得留作未说明的空值：

| 字段 | 含义 |
|---|---|
| `parameter_id` | 稳定的参数标识，不以块路径或中文显示名代替 |
| `value` / `status` | 数值、表格、函数引用或明确占位状态 |
| `unit` | SI 单位或明确的无量纲定义 |
| `source_tag` | `OFFICIAL`、`LITERATURE`、`ENGINEERING_DEFAULT`、`PLATFORM_DEFAULT`、`PLACEHOLDER`、`HISTORICAL_CASE` 或 `L0.5_SCREENING` |
| `source_ref` | 文件、数据手册页码、文献标识、模型版本或测试记录 |
| `identity_scope` | 具体设备、平台默认对象或历史案例身份 |
| `validity_range` | 温度、SOC、电压、电流、频率、时间窗口、冷却或机械条件 |
| `confidence` | 已验证、来源可信未验证、工程假设或未知 |
| `replace_impact` | 替换该参数后必须重新检查的接口、派生参数和验证项 |
| `verification_status` | 未验证、结构读回、文献比对、台架比对或其他明确状态 |

### 10.3 各配置的最小参数集

| 配置 | 必填参数类别 | 不允许的替代 |
|---|---|---|
| BatteryConfig | 拓扑、容量语义、OCV/阻抗或等效电路、初始 SOC、热容/热阻、允许 V/I/SOC/T 包线、冷却接口 | 用额定容量直接代替允许脉冲电流；用 DC 电阻代替所有频率下阻抗而不说明适用性 |
| HVPathConfig | 连接拓扑、开断/预充语义、R/L/C、传感位置、允许 V/I、压降和损耗口径 | 用电池端电压代替 DC-link 电压；忽略储能或线束损耗 |
| InverterConfig | DC 电压范围、相电流/母线电流能力、器件/平均值模型、调制限制、损耗模型、热边界 | 用控制器软件元数据代替器件电压/电流额定值；将模型开关频率当设备能力 |
| MotorConfig | 电气参数、相电流能力、机械锁止/惯量、转速边界、热参数与温度限制 | 用控制器目标电流代替电机相电流能力；将零速设定视为锁止边界已验证 |
| ControlConfig | `CandidatePath_ID`、参考坐标/物理语义、波形/请求定义、反馈量、采样/控制/PWM 时序、饱和、抗饱和、速率转换、执行使能 | 用理想电流源代替控制可实现性；用连续信号掩盖采样或死区 |
| ThermalConfig | 环境、冷却、热容、热阻、热源映射、测点和温度限制 | 用末时刻温度代替窗口热行为；将未知热边界设为零散热而不标注 |
| ProtectionConfig | 硬/软限制、来源、优先级、延时、去抖、复位与失效状态 | 用 `NaN`、零或任意常数表示“未触发”；在无来源时默认允许 Active |
| ScenarioConfig | 初始状态、请求、驻车状态、外部条件、持续时间、记录窗口 | 复用历史工况数值而不重新标注其来源和目的 |

### 10.4 派生参数与一致性规则

1. 所有派生量必须引用其上游参数和公式或模型版本。例如，电池包总电阻、能力限值、热源、电流基值和 DC-link 允许范围均不得以孤立常数存在。
2. 电池拓扑、容量、并联数、等效阻抗、允许电流和热容必须来自同一 BatteryConfig；不允许混用不同电池身份或拓扑的值。
3. 逆变器 DC 电压、相电流、母线电流、开关频率、采样量程、调制限制、损耗和热边界必须通过同一 InverterConfig 与 ControlConfig 的兼容性检查。
4. PMSM 电气参数、相电流限制、磁链、机械锁止、热容和温度边界必须来自同一 MotorConfig 或明确的跨来源适配说明。
5. `ScenarioConfig` 只能请求候选命令，不能绕过 BatteryConfig、InverterConfig、MotorConfig、ProtectionConfig 的限制。
6. 各配置的单位、符号、时间基准和 RMS/能量窗口必须满足 G1；不兼容的组合必须在 Precheck 拒绝。

### 10.5 组合兼容性检查

每个 SystemConfig 至少完成下表检查，并记录 `pass`、`fail`、`unknown` 或 `not_applicable`，不得以空白代替结果：

| 检查域 | 最低检查内容 | 未通过或未知时的处理 |
|---|---|---|
| 电压 | 电池可用电压、DC-link 范围、逆变器范围、控制测量范围和调制需求 | Precheck 拒绝 Active，或限制为明确研究工况 |
| 电流 | 电池支路、母线、器件、相电流、传感和热等效电流的峰值/平均/RMS 包线 | Precheck 拒绝 Active |
| 功率与能量 | 端口功率方向、DC-link 储能、损耗和能量窗口定义 | 只能输出 C0/C1，不能给出兼容性推断 |
| 热 | 电池、电驱、冷却和环境热边界的来源与适用范围 | 禁止输出热风险或温升 C2 结论 |
| 控制与时序 | 采样、PWM、死区、最小脉宽、反馈延时、饱和与保护更新关系 | 禁止输出电流可实现性 C2 结论 |
| 机械 | 驻车/锁止、转速和反作用载荷边界 | 禁止将零速结果解释为实际驻车可实现 |
| 保护 | BMS、硬件、热和监督逻辑的阈值、优先级、可恢复性 | Precheck 拒绝 Active |
| 数据与来源 | 参数身份、来源、适用范围和验证状态 | 降低结论等级并输出数据缺口 |

### 10.6 Precheck 完整性门槛

Precheck 分为三类结果：

| 结果 | 条件 | 允许的后续行为 |
|---|---|---|
| `ready_for_active` | 电压、电流、保护、关键时序和配置身份均通过；热/能量审计输入完整 | 允许进入 Active，并按配置结论等级输出 |
| `research_limited` | 非关键参数为明确 `PLACEHOLDER`，且研究范围、限制和结论上限已定义 | 仅允许明确标记的研究工况；不得输出被缺失项覆盖的 C2 结论 |
| `blocked` | 关键来源/单位/身份缺失，或电压/电流/保护/时序检查失败或未知 | 禁止 Active，输出阻断项和所需数据 |

下列字段在首次 M3 Active 工况前属于关键字段：电池到 DC-link 的拓扑和正方向、所有端口电压/电流量纲、候选命令的限幅链、逆变器与电机的电流/电压能力身份、采样/PWM/保护时序关系、RMS 与能量记录窗口、驻车机械边界、保护退出路径。

### 10.6.1 Precheck 元数据输入契约

首轮配置 Precheck 采用以下最小元数据结构。它验证配置登记、占位阻断和兼容性声明，不以孤立的 `complete` 标记替代参数数值、来源和物理验证。

| 层级 | 必填字段 | 规则 |
|---|---|---|
| SystemConfig | `systemConfigId`、`isContractBaseline`、`candidatePathId`、`components`、`compatibility`、`optionalDataGaps`、`researchScope` | 契约基线或空/`PLACEHOLDER` 候选路径必须阻断；`optionalDataGaps` 必须显式给出空集或限制项；存在限制项时必须声明可运行研究范围与禁止外推内容 |
| 每个 component | `id`、`status`、`sourceTag`、`identityScope`、`requiredParametersComplete` | `id`、身份或来源为 `PLACEHOLDER`，或必要参数未完成，均阻断 Active |
| Control component | `candidatePathId` | 必须与 SystemConfig 的 `candidatePathId` 一致，且不得为 `PLACEHOLDER` |
| compatibility | `voltage`、`current`、`powerEnergy`、`thermal`、`controlTiming`、`mechanical`、`protection`、`dataSource` | 首次 Active 的八项状态均须为 `pass`；`fail`、`unknown` 和 `not_applicable` 均阻断 |

`requiredParametersComplete=true` 只能由已登记且可追溯的参数集产生；实现中应将其与参数注册表或数据字典交叉核验。当前元数据 Precheck 不验证参数量值、方程或物理兼容性本身，因此即使返回 `ready_for_active`，后续仍必须执行 V0-V4 以及对应的物理边界检查。`research_limited` 只在 `researchScope` 明确时允许进入对应研究工况；该范围外仍视为禁止 Active。

### 10.7 平台默认配置的使用边界

`PLATFORM_DEFAULT` 只能用于验证 M3 结构、接口、配置替换和约束审计流程。它必须：

1. 以独立 `SystemConfig_ID` 登记，不能伪装为具体设备。
2. 不使用 M2/V4-I 的未闭合容量、限流、拓扑、电阻、频率或控制判据作为默认值。
3. 为每一关键参数保留替换入口和来源说明。
4. 在结论中明确其适用范围为平台工程兼容性研究，不能外推为产品性能或安全边界。

### 10.7.1 首轮平台契约基线

首轮平台基线命名为 `M3-PD-CONTRACT-v01`。它是配置结构、接口、Precheck 和验证工况的治理基线，不是也不得登记为具体电池包-电驱模块的 `SystemConfig_ID`。

| 项目 | `M3-PD-CONTRACT-v01` 的状态 | 允许行为 |
|---|---|---|
| 系统目标 | `PLATFORM_DEFAULT`，兼容性优先 | 定义约束裕度、主导限制、数据缺口和配置替换审计 |
| 候选路径 | `PLACEHOLDER` | 可验证路径字段缺失时 Precheck 阻断；不得猜测 d/q 轴、相电流、频率或波形 |
| 电池/高压/电驱能力 | `PLACEHOLDER`，待具有身份和适用范围的配置填入 | 不得进入 Active；可执行配置阻断、接口和指标独立测试 |
| 主线电驱表达 | `PLATFORM_DEFAULT`：平均值逆变器、离散控制、DC-link 动态、PMSM/热接口 | 仅当实际子配置完整时用于系统兼容性主线 |
| PWM 复核 | `PLATFORM_DEFAULT`：代表工况的可选验证层 | 仅验证平均值模型未覆盖的纹波、死区、时序或损耗假设，不作为主线配置替代 |
| 证据状态 | `OFFICIAL`/`LITERATURE`/`PLACEHOLDER` 的受控组合 | C0-C2 受限推断；C3 等待独立数据验证 |

第一个可进入 Active 的系统组合必须另行生成唯一 `SystemConfig_ID`，并以完整的 BatteryConfig、HVPathConfig、InverterConfig、MotorConfig、ControlConfig、ThermalConfig、ProtectionConfig 和 ScenarioConfig 替代上述关键占位项。`M3-PD-CONTRACT-v01` 本身只能支持 `SC-00_ConfigBlocked`、`SC-02_MetricOracle` 及不要求 Active 的结构/Idle 检查。

### 10.8 G2 通过条件

G2 通过需确认：

1. 配置对象、ID 规则、最小参数字段和组合规则可以支持单模块及后续多模块扩展。
2. 所有关键参数的来源、身份、适用范围、替换影响和验证状态已有登记位置。
3. Precheck 的 `ready_for_active`、`research_limited`、`blocked` 行为明确，关键未知项不会默认通过。
4. 平台默认配置与未来具体设备配置的使用边界明确，未继承 M2 或历史案例的未闭合参数。
5. G3 可以以配置和接口为索引定义验证证据与验收条件。

## 11. G3 验证与验收计划

状态：既有 G3 草案基线，待阶段 4 根据共同确认的顶层架构、优先级和验收重新组织；当前不创建验证工况或运行验证。

### 11.1 验证层级与结论映射

| 验证层级 | 目的 | 最低证据 | 可支持的最高结论 |
|---|---|---|---|
| V0 配置与结构 | 验证 SystemConfig、端口、单位和信号流完整 | 配置完整性记录、模型读回、结构检查 | C0 |
| V1 数值口径 | 验证 RMS、能量、损耗、热与状态指标可按定义计算 | 已保存时间基准、窗口、公式/模型版本和单位检查 | C1 |
| V2 受限行为 | 验证命令、限幅、保护、损耗和热路径在明确假设下的响应 | 受控工况、事件记录、边界状态和审计输出 | C2，但仅限通过的接口与假设范围 |
| V3 数值鲁棒性 | 验证结果不依赖单一 solver、步长、容差或初值偶然成立 | 交叉求解、步长/容差/初值比较和误差记录 | C2 的数值可信度支撑 |
| V4 参数与配置 | 验证参数替换、组合检查和敏感性结论可追溯 | 配置替换记录、兼容性矩阵、敏感性结果 | C2 的适用范围与主导因素 |
| V5 独立数据 | 验证模型对台架、供应商或实测数据的适用范围与误差 | 数据版本、对齐方法、误差指标和未覆盖区域 | C3 |

未通过或未执行的更高层验证不否定较低层事实，但必须限制结论等级和使用范围。

### 11.2 V0 配置与结构验收

| 检查项 | 通过条件 | 必须记录 | 失败处置 |
|---|---|---|---|
| 配置完整性 | SystemConfig 包含全部子配置 ID、版本、来源状态和兼容性检查结果 | SystemConfig 清单与 Precheck 结果 | `blocked`，禁止 Active |
| 接口完整性 | I0-I7 均有生产者、消费者、单位、方向、有效性和数据类型定义 | 接口矩阵、模型读回或等价证据 | 修复定义或结构，不以 Terminator/常数掩盖 |
| 物理网络 | 各物理网络具备必要参考、求解配置和边界；无未解释结构 error | `model_check`、读回和保留理由 | 结构 error 必须修复；warning 须逐条解释和回归 |
| 参数可追溯 | 关键参数能回溯到 G2 元数据，派生量能回溯到公式/上游参数 | 参数注册表、配置 ID、派生关系 | `blocked` 或降为研究限定 |
| 模式安全 | 无效配置/输入/传感和关键限制未知时不进入 Active | Precheck 与模式转换记录 | 修复逻辑或阻止场景运行 |

### 11.3 V1 RMS、功率与能量验收

| 检查项 | 通过条件 | 最低验证方式 |
|---|---|---|
| RMS 语义 | 电池支路、母线、相电流和热等效量分别按 G1 的窗口积分定义计算 | 用可解析测试信号或独立后处理交叉验证；记录 `Tw`、采样和积分误差 |
| 峰值/平均/RMS 区分 | 同一信号的峰值、有符号平均和 RMS 均可独立输出且不互相复用 | 单位与量级检查、脉冲和零命令边界工况 |
| 端口功率 | 电池、DC-link、逆变器、PMSM 和机械端口的 V/I/P 方向一致 | 符号翻转、零功率和已知方向工况 |
| 储能与损耗 | DC-link、电机机械、热网络和必要电池内部储能/损耗项被显式列出 | 同一窗口的能量积分和储能差分 |
| 能量残差 | 仅在端口、储能和损耗齐全且同时间窗口下报告；残差包含绝对值、相对值和未包含项 | 能量账本；禁止以“无功”或“未建模”笼统解释残差 |
| 热源映射 | 每一发热量有电流口径、阻抗/损耗来源和热网络去向 | 电-热功率一致性、零电流和已知损耗边界 |

V1 通过不要求能量残差已达到工程容差，但必须能解释账本范围、残差定义和未纳入项；没有完整账本时只能报告“能量审计未完成”。

### 11.4 V2 命令、控制、限制与保护验收

| 场景 | 通过条件 | 观察量 |
|---|---|---|
| Idle/零命令 | 无有效脉冲命令；不出现未解释的持续电流、损耗或温升 | 命令、实际电流、功率、模式和保护状态 |
| Precheck 通过 | 配置和关键限制完整时才允许进入 Active | 配置 ID、限制来源、模式事件 |
| Precheck 阻断 | 关键未知、单位/方向错误或兼容性失败时拒绝 Active | 阻断原因、模式和输出安全状态 |
| 候选命令跟踪 | 在明确控制策略和有效工况下，实际电流、调制裕度和饱和行为可审计 | `i_ref`、`i_meas`、误差、调制、Vdc、采样时标 |
| 限幅 | 电池、母线、逆变器、电机或热限制触发时，命令按优先级降额或停机 | 约束来源、触发时刻、限幅前后命令、实际响应 |
| 保护 | 硬限制、故障或关键传感无效时进入 Stop/Fault；复位后必须重新 Precheck | 事件顺序、延时、复位条件和安全状态 |
| 驻车边界 | 零速或锁止以明确机械边界实现，机械功率和反作用载荷解释一致 | `omega_m`、`T_e`、`P_mech`、锁止状态 |

控制跟踪的具体误差、饱和和保护延时容差必须来自 ControlConfig、ProtectionConfig 或明确的工程研究假设。未定义容差时，只能报告响应事实与数据缺口，不能标记“控制通过”。

### 11.5 V2 热行为验收

| 场景 | 通过条件 | 结论限制 |
|---|---|---|
| 静态热平衡 | 无发热或恒定边界下，温度与热流行为符合已定义边界 | 仅验证热网络和初值，不验证产品散热能力 |
| 受控脉冲加热 | 电池、电驱和冷却的温升/热流可追溯到相应热源和边界 | 只在热参数来源和适用范围内输出 C2 推断 |
| 热限制/降额 | 温度或热裕度触发时，监督层按配置进入 Derate/Stop | 未经数据验证不得输出器件安全许可 |
| 冷却边界变化 | 冷却条件变化产生方向和量级可解释的热响应 | 冷却模型未验证时仅用于敏感性或风险识别 |

温升趋势、末时刻温度或单个热测点不等于热模型精度。热模型的 C3 结论必须基于独立温度/热流数据及明确误差指标。

### 11.6 V3 求解器与数值收敛验收

| 检查项 | 通过条件 | 记录内容 |
|---|---|---|
| 初值一致性 | 同一配置和工况的初始状态完整、可重放且无隐式工作区依赖 | 初值来源、配置 ID、初始化日志 |
| 步长收敛 | 降低最大步长或固定步长后，关键 KPI、RMS、能量和事件时刻变化满足预定义容差 | 步长组、KPI 差异、运行时间 |
| 容差收敛 | 收紧相对/绝对容差后，关键结果稳定或误差被量化 | 容差组、误差和运行时间 |
| 求解器交叉 | 对适用的替代求解器或离散化策略，结果差异可解释 | 求解器设置、适用范围、差异说明 |
| 事件鲁棒性 | 限幅、保护和模式转换不因微小步长/容差变化发生无解释的事件顺序改变 | 事件序列、时间戳、状态转换 |

G3 不预设数值容差。G2/ControlConfig/ScenarioConfig 应针对控制、热和能量 KPI 分别定义可接受误差与运行成本；未做 V3 时，任何长时或高频结果均须标为数值未收敛。

### 11.7 V4 参数替换、组合与敏感性验收

| 检查项 | 通过条件 | 最低证据 |
|---|---|---|
| 配置替换 | 替换任一子配置后，受影响的接口和派生参数被重新检查 | 新旧 SystemConfig、差异与兼容性矩阵 |
| 配置隔离 | 替换不影响无关配置；不存在隐式共享参数或历史工作区污染 | 参数追踪、模型读回、回归工况 |
| 单参数敏感性 | 关键 KPI 对来源不确定参数的变化方向和范围可解释 | 扫描范围、来源、KPI 和结论限制 |
| 组合敏感性 | 关键配置组合的兼容性差异可追溯 | 组合矩阵、被拒绝/允许原因 |
| 数据缺口影响 | 对结论影响最大的未知参数、接口或试验需求被明确排序 | 风险/数据缺口清单 |

敏感性分析不是参数寻优，也不能用来将缺失设备数据拟合成“真实产品参数”。

### 11.8 V5 独立数据验证验收

V5 仅在获得明确授权和可追溯数据后执行。每类数据验证必须先固定：数据对象与版本、工况、传感器位置、采样与滤波、时间同步、参数是否参与拟合、验证集是否独立、误差指标和可接受范围。

| 数据类别 | 可验证对象 | 不可替代的内容 |
|---|---|---|
| 电池 HPPC/EIS/脉冲测试 | OCV/阻抗、端电压、发热与 SOC 变化适用范围 | 析锂或寿命结论仍需专门证据 |
| 逆变器/电驱台架 | 电流跟踪、损耗、热、调制与保护时序 | 不能替代电池低温数据 |
| 热台架 | 温度、热流、热阻、热容和冷却响应 | 不能替代器件电流能力证明 |
| 整机或车辆日志 | 使能、状态机、母线和系统边界 | 不能替代未测量的内部状态或寿命结论 |

拟合数据与验证数据必须分离。C3 结论必须同时报告误差指标、未覆盖工况、参数不确定性和外推限制。

### 11.9 首轮 V0-V4 最小工况包

首轮工况包以配置派生的边界和请求定义，不预填历史案例的电压、电流、频率、容量或持续时间。每个工况必须指定一个 `SystemConfig_ID`、一个 ScenarioConfig、输入数据版本和预期模式序列；没有 `ready_for_active` 或 `research_limited` Precheck 结果时，不得运行需要 Active 的工况。

| 工况 ID | 覆盖层级 | 前置条件与刺激 | 最低通过条件 | 必要审计输出 |
|---|---|---|---|---|
| SC-00_ConfigBlocked | V0 | 缺失一项关键单位、身份、限幅链或保护退出路径 | Precheck 返回 `blocked`，始终不进入 Active | 阻断字段、配置 ID、模式序列、安全输出 |
| SC-01_IdleBaseline | V0/V1/V2 | 有效配置、零加热请求、明确初始热边界 | 保持 Idle；无未解释持续电流、端口能量或热源 | 命令、模式、V/I/P、热源、温度、RMS 与能量窗口 |
| SC-02_MetricOracle | V1 | 可解析的零、常值、正负对称和脉冲测试信号；不要求电驱 Active | 峰值、有符号平均、窗口 RMS、积分能量与独立解析值一致；窗口边界可复现 | 信号定义、解析期望值、计算结果、`Tw`、误差和积分方法 |
| SC-03_PrecheckActive | V0/V2 | 所有关键配置兼容；候选请求在配置允许范围内 | 仅在 Precheck 成功后进入 Active；限制来源和模式转换完整 | Precheck、配置兼容性、模式事件、使能和限幅链 |
| SC-04_CommandLimit | V1/V2 | 分别施加高于电池、母线、逆变器、电机或热边界的候选请求 | 激活最高优先级有效限制；实际响应不绕过限制 | 请求、限幅前后命令、约束来源、实际 V/I/P、事件时刻 |
| SC-05_FaultExit | V0/V2 | 注入关键传感无效、硬件限制或外部禁止 | 按保护优先级进入 Stop/Fault；复位后重新 Precheck | 故障源、保护状态、模式事件、复位条件、安全输出 |
| SC-06_EnergyLedger | V1/V2 | 在有效 Active 或明确研究限定的脉冲窗口内运行 | 所有端口、储能、损耗和未纳入项使用同一窗口入账；残差可复算 | 端口能量、储能差分、损耗、残差绝对/相对值、账本范围 |
| SC-07_ThermalBoundary | V1/V2 | 零热源、恒定热源和冷却边界变化三类热场景 | 热流方向、稳态/瞬态趋势与配置边界一致；热限制可触发 | 热源映射、温度、热流、冷却边界、Derate/Stop 事件 |
| SC-08_ParkConstraint | V0/V2 | 明确锁止或允许转动的机械边界下请求加热 | 零速、转矩、机械功率和反作用载荷的语义一致；不得把未定义锁止当作驻车通过 | 锁止状态、`omega_m`、`T_e`、`P_mech`、模式和告警 |
| SC-09_NumericalRobustness | V3 | 选定一个已通过 SC-03 至 SC-08 的代表工况，改变步长、容差、初值或适用求解器 | 关键 KPI 和事件序列满足配置定义的数值容差，或差异被量化并阻断相应结论 | 求解器组、初值、运行设置、KPI 差异、事件时间差、运行时间 |
| SC-10_ConfigReplacement | V4 | 在保持 ScenarioConfig 不变时替换一个子配置或关键不确定参数 | 新组合完成 Precheck；影响范围准确；无关配置和结果不被隐式改变 | 新旧配置差异、派生参数、兼容性矩阵、KPI 差异、数据缺口排序 |

`SC-02_MetricOracle` 是 V1 的独立指标测试，不应用候选物理模型输出自身验证 RMS 或能量算法。`SC-04_CommandLimit` 和 `SC-05_FaultExit` 可以共享模型结构，但必须作为不同验证记录保存，以防限幅行为掩盖保护退出缺陷。

### 11.10 验收记录与回归资产

1. 每次验证记录必须关联 `SystemConfig_ID`、ScenarioConfig、模型/脚本版本、求解器设置、时间窗口、输入数据版本和输出结论等级。
2. 回归资产只保留能验证已冻结接口、配置或风险的最小必要工况；不默认保存完整 timeseries、重复结果或大规模扫描。
3. 结构检查、配置检查、RMS/能量审计、控制/保护、热、收敛和数据验证应分别报告，通过项不能掩盖未执行项。
4. 任何 warning、失败、未知参数或数据缺口均须在验证记录中给出处理状态：修复、保留理由、阻断或后续数据需求。

### 11.11 G3 通过条件

G3 通过需确认：

1. V0-V5 的验证层级、最小证据、结论上限和未通过处置可覆盖 G0 的工程问题与 G1 的 I0-I7 接口。
2. RMS、能量、热、控制、保护、收敛、参数替换和独立数据验证均有独立口径，不能以单次仿真替代。
3. 每项未来 M3 结论都能追溯到配置、工况、验证层级和结论等级。
4. 建模实现开始前，首轮 V0-V4 工况包、观察量、前置条件与通过条件已定义；V5 明确为有数据后执行的独立门槛。
5. G0-G3 一致后，才可决定 M3 模型创建方式和最小实现顺序。

| 门槛 | 状态 | 已记录的依据 |

## 13. 已确认路线与待获取数据

已确认：M3 以单电池包-单电驱模块为最小可验证单元，并通过接口与配置层预留多模块扩展能力。

| 已确认项 | 采用结果 |
|---|---|
| 首轮工程目标 | 兼容性优先；识别主导限制和裕度，不做产品性能寻优 |
| 候选电流路径 | 接口化、可替换；在获得受控配置前不预设 d/q 策略、相电流、频率或波形 |
| 电驱保真度 | 平均值主线加 PWM 代表工况抽查 |
| 证据路线 | 官方结构/接口与文献量级先行；C3 等待可追溯独立台架、供应商或实测数据 |

仍需按具体系统组合获取，而非由平台默认猜测的数据：电池低温 OCV/阻抗、能力和热边界；高压路径与 DC-link 参数；逆变器与 PMSM 电压/电流/热能力；候选路径与驻车机械约束；控制/保护时序和阈值；独立验证数据及其适用工况。
