| 决策项 | 选择 | 理由 |

### 1.2 二阶 ECM 频域阻抗（NPBH 式5）

```
Re(Z) = R0 + R1/(1+(2πf)²R1²C1²) + R2/(1+(2πf)²R2²C2²)
Re(Z_dc) = R0 + R1 + R2  (f=0)
```

### 1.3 频域热生成（Zhang 式1）

```
Q̇ = I_RMS² · Re(Z)
```

### 3.3 结果分析

1. **频域 vs 时域产热对比**：P_harmonic（路径A：固定窗口 DFT 谐波分解）与 P_td（路径B：时域 I²×Re(Z_dc)）在同一量级但数值不同，符合预期--频域模型对各频率点做 DFT 后按 Re(Z,f) 加权累加，时域模型用 Re(Z_dc) 逐点计算。freq_time_residual（P_harmonic - P_td）在所有工况中有限（-0.13~+0.01W），可通过波形等效校核归因。bn=0 时残差为正（谐波分解捕获了 DC 外的交流分量），bn>0 时残差为负（2kHz PWM 分量在频域 Re(Z) 较低，而时域用全频段 Re(Z_dc) 高估）。

2. **P_harmonic 独立 DFT 复算**：运行器对 [0.03,0.04]s 的 5000 个采样点做独立离线 DFT，与模型内 DFT 结果逐工况比较，最大差 8.8e-05W（3.1-D），远低于 1e-2 阈值。这证明模型内固定窗口 DFT 实现正确。

3. **M4 固定 R_batt 对比**：P_M4（使用 R_batt_heat_ref=M401C_R_batt_heat_ref_Ohm）显著高于频域和时域估算（P_M4 在 bn=0.3 Id=10 时 1.64W vs P_harmonic 0.088W），因为 M4 的 R_batt_heat_ref 是高温参考值，不反映低温频域阻抗。这证实了取舍清单中"先与二阶 ECM 频域阻抗并列审计，再决定替换"的必要性。

4. **bn 的影响**：bn 从 0 到 0.5（Id=10A），死区采样从 0 增到 2164，P_harmonic 从 0.016W 增到 0.083W，证实 dSVPWM 换流增强了电池侧交流电流和频域产热。

5. **Id_ref 的影响**：bn=0.3 下 Id_ref 从 10A 到 20A，死区采样从 1446 变到 1473（bn 相同死区占比相近），P_harmonic 从 0.088W 增到 0.435W，P_M4 从 1.64W 增到 4.71W，证实电流幅值增大显著提升产热。

6. **死区段**：bn=0 时无死区，bn>0 时死区段出现（1446~2164 个采样点），与 M5-02 验证一致（bn 越大死区占比越高）。

## 6. 已知事项

1. **ECM 参数占位**：R0/R1/C1/R2/C2 用 ENGINEERING_DEFAULT 占位，不代表任何真实电芯。所有频域热结果标 PlatformEstimate，不形成产品温升结论。未来如有目标电芯 EIS 数据，应替换参数并标 LITERATURE。
2. **温度/SOC 依赖未实现**：当前 Re(Z) 不随 T/SOC 变化（平台假设）。T_pack 和 SOC 已作为输入接口预留（blk_930 接受 T_pack/SOC 但当前不修改参数），未来可扩展为查表。
3. **固定窗口 DFT 滞后**：MATLAB Function 块的 latched 输出在窗口完成后的下一采样点才更新，因此 StopTime 须设为 Window_Stop + RMS_Window_s（0.05s）才能读到评价窗口 [0.03,0.04]s 的结果。受控运行器已固化此设置。
4. **P_M4 与频域估算的差异**：P_M4（R_batt=M401C_R_batt_heat_ref_Ohm，高温参考值）显著高于频域估算，证实了频域模型的必要性；两者口径不同，不代表 M4 固定 R_batt "错误"。
5. **P_freq 与 P_harmonic 口径差异**：P_freq 用 I_rms²×mean(Re(Z))（简化频域估算），P_harmonic 用固定窗口 DFT 各频率分量加权 Re(Z)（精确谐波分解）。两者并列审计，P_harmonic 是路径A 的权威值。
6. **短时窗口限制**：0.04s 评价窗口无法看到实际温升。频域-时域交叉复算验证通过后，如需温度演化证据，可增加长时窗口工况。

## 7. 结论边界

1. v2 模型已实现频域电热模型（二阶 ECM + 频域热生成 + 谐波分解 + 时域交叉校核），四工况验证全部通过（CSV 已写入 `04_仿真结果/M5-03_freq_thermal_ledger_v01.csv`）。这是 Reference/Platform 诊断证据，不是设备加热性能或温升结论。

3. ECM 参数为 ENGINEERING_DEFAULT 占位，所有频域热结果标 PlatformEstimate，不形成产品温升结论。
4. 未实现析锂安全判据--属于 M5-04。
5. 未实现控制对照和拓扑替代--属于 M5-05。

## 8. 阻塞点排查与经验积累

本节记录 M5-03 收口过程中实际遇到的三个阻塞点、排查路径和最终解决方案，作为后续建模任务的工作流强化依据。**这些阻塞点在初次执行记录中未被捕获，因为初次记录是在验证运行实际执行前基于设计意图写就的假设值。**

### 8.1 阻塞点 1：执行记录先于验证运行写入（流程性阻塞）

**现象**：接手 M5-03 收口时，发现执行记录第 3 节已包含完整的验证结果表，但 `04_仿真结果/` 目录下仅有 `.gitkeep`，CSV 不存在。顶层规划仍标记 M5-03 为"未启动"。

**排查**：通过 `glob` 确认 CSV 缺失；通过 `model_overview` 确认模型中频域块（blk_930-934）已存在且结构完整；通过 `model_check` 确认 blk_843 healthy。结论：模型结构修改已完成并保存，但受控运行器从未实际执行过验证工况，执行记录中的数值是设计阶段的预估值。

**解决**：以实际执行为准，推翻假设值，重新运行全部 4 工况生成真实 CSV，再据此回填执行记录。

**经验教训**：
- **执行记录的验证结果必须来自实际运行的 CSV，不能在设计阶段预写假设数值。** 这是 AGENTS.md 第 5.4 节"脚本通过或仿真完成不等于物理正确"的延伸：未运行的脚本连"脚本通过"都不是。
- 收口一个阶段时，第一步必须验证"声称已完成的产物是否真实存在"（CSV 文件、logsout 信号、块 ID 读回），而不是直接信任执行记录的文字。
- 后续阶段接手时，应建立"产物存在性核验"清单：CSV 是否存在且非空、块 ID 是否与记录一致、workspace 参数是否就位、logsout 信号数是否匹配。

### 8.2 阻塞点 2：RMS_Window_Complete 瞬时标志的断言逻辑错误

**现象**：受控运行器首次执行（preflight）即失败，报错 "The fixed RMS window did not complete."

**排查**：逐样本读取 `RMS_Window_Complete` 信号，发现该标志是瞬时的--仅在 10ms 窗口边界处（t=0.01/0.02/0.03s）为 1，其余时间为 0。在 StopTime=0.04s 时，末样本恰好落在两个窗口边界之间，标志为 0。但 SystemKPI(44)=1、SystemKPI(45)=1 表明账本实际有效。原 runner 的 `assert(RMS_WindowComplete_Final == 1)` 检查的是末样本值，而非"窗口是否曾完成"。

**解决**：新增 `RMS_WindowComplete_Any = max(wcFlag) > 0.5` 字段，断言改为检查 `Any` 而非 `Final`。保留 `Final` 字段供审计。

**经验教训**：
- **瞬时标志（pulse/edge 类型）不能用作"是否曾发生"的判据，除非取 max 或累计计数。** Simulink 中很多窗口完成标志、触发标志是电平型脉冲，只在事件发生的那一拍为 1。
- 编写受控运行器的断言前，必须先用 `diagnose` 模式对单个工况做逐样本信号检查，确认标志的时序语义（持续型 vs 瞬时型），再决定断言策略。
- M4 运行器（runM402B/runM403B）通过 `lastKpiValue(kpi, 27)` 读取 SystemKPI 中的窗口完成字段，而非直接读 logsout 的瞬时信号，避开了此问题。M5-03 runner 直接读 logsout 信号时暴露了这一差异。

### 8.3 阻塞点 3：固定窗口 DFT 的一窗口输出滞后（核心阻塞）

**现象**：4 工况全部运行完成后，3.1-D（bn=0.3, Id=20A）在 DFT 一致性断言处失败：`P_harmonic_DFT_Diff_W = 0.182 > 1e-2 阈值`。模型 P_harmonic=0.253W，独立 DFT=0.435W。

**排查过程（最耗时）**：
1. 初步怀疑独立 DFT 公式与模型不一致 -> 逐项对比公式（phase、amplitude、weight），完全相同。
2. 怀疑循环缓冲区读取顺序导致样本错位 -> 验证 circshift 后 DFT 幅值不变（2kHz 是窗口周期的整数倍，移位不改变幅值），排除。
3. 怀疑模型 `f_eval` 参数未正确传入 -> 确认 Stateflow Parameter Scope=Parameter, CompiledSize=[1,7]，7 个频率全部就位。
4. 怀疑 2kHz 分量被模型遗漏 -> 分解独立 DFT 各频率贡献，发现 2kHz 贡献 0.214W，模型结果(0.253) ≈ 独立 DFT(0.435) - 2kHz 贡献(0.214) + 小差异。模型确实缺少 2kHz 项。
5. **关键突破**：检查 P_harmonic 的完整时间序列，发现模型在 t=0.04s 输出 0.253，但这对应的是窗口 [0.02,0.03) 的结果（独立 DFT [0.02,0.03)=0.253），而非评价窗口 [0.03,0.04) 的结果（独立 DFT=0.435）。模型输出存在一整个窗口（10ms）的滞后。

**根因**：MATLAB Function（Stateflow EMChart）块的 `persistent` 变量在 `sample_count >= N_WINDOW` 时计算并 latch 结果，但 latch 后的输出在当前采样点写入，而 logsout 记录的同一采样点可能尚未反映该更新（取决于块的执行顺序和输出端口的采样时序）。实际表现为：窗口 [0.03,0.04) 在 t=0.04s（第 20000 个样本）完成计算，但 latch 值要到 t=0.05s（下一个采样点）才出现在 logsout 中。

**解决**：StopTime 从 0.04s 改为 0.05s（= Window_Stop + RMS_Window_s = 0.04 + 0.01），使末样本对应评价窗口 [0.03,0.04) 的 latch 结果。验证：StopTime=0.05s 时模型 P_harmonic=0.4347W，与独立 DFT 0.4348W 一致（差 8.8e-05W）。

**经验教训**：
- **MATLAB Function 块中使用 persistent 变量做固定窗口累积时，latched 输出有一拍（或一窗口）滞后。** 评价窗口 [T_start, T_stop] 的结果在 T_stop 时刻完成计算，但可能要到 T_stop + 一个采样周期（或一个窗口周期）才出现在输出日志中。
- **受控运行器的 StopTime 不能等于评价窗口的结束时间，必须留出输出传播余量。** 对于 10ms 固定窗口，StopTime = Window_Stop + RMS_Window_s 是安全设置。此规则应写入 AGENTS.md 第 11 节的工况定义标准。
- 当模型值与独立复算值出现系统性偏差时，第一步应检查"两者是否对应同一时间窗口"，而非直接怀疑公式实现。通过打印模型输出的完整时间序列（distinct values + first occurrence time），可以快速定位窗口对齐问题。
- M4 运行器（StopTime=0.08s, 评价窗口 [0.04,0.08]s）未遇到此问题，因为 M4 的账本块（blk_888/blk_897）使用不同的 latch 机制（在窗口边界直接输出，无额外滞后）。M5-03 的频域块（blk_932）使用 persistent + sample_count 重置机制，滞后行为不同。不同块的 latch 语义不能互相假设。

### 8.4 工作流强化建议（适用于后续 M5-04/M5-05 及类似阶段）

1. **产物存在性先行核验**：接手任何阶段收口时，先验证声称的产物（CSV、块 ID、参数、信号数）是否真实存在且与记录一致，再开始任何修改或验证工作。
2. **单工况 diagnose 先行**：全量运行前，先用 `diagnose:scenario_id` 模式跑一个工况，逐样本检查关键信号的时序语义（持续型 vs 瞬时型、latch 滞后），确认断言逻辑正确后再全量运行。
3. **StopTime 留输出余量**：涉及 persistent/latch 固定窗口的块，StopTime = Window_Stop + Window_Period，确保末样本反映评价窗口结果。
4. **执行记录与验证运行同步**：验证结果必须从实际 CSV 回填，不在运行前预写假设数值。执行记录的"完成"以 CSV 存在且断言通过为准。
5. **窗口对齐优先于公式排查**：模型值与独立复算值不一致时，先确认两者对应同一时间窗口（打印时间序列的 distinct values 和 first occurrence time），再排查公式实现。
