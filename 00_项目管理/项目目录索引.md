## 使用规则

1. 先读取 Active 文件，再决定是否调用归档证据。
2. 不将归档中的模型、脚本、计划、参数和"下一步"文字视为 M3 指令。
3. M3 的顶层目标、架构、路径、里程碑和验收以 `M3_顶层建模规划_v02.md` 冻结版为准。
4. 当前阶段不把归档或独立的 G0-G3、Precheck、RMS 脚本或固定实施顺序视为已确认的执行主线。
5. `M3-PD-CONTRACT-v01` 仍不可运行；M4-02B、M4-03B 运行器和各自唯一 CSV 是受控的 Reference / Platform 资产，不得被当作产品配置或完整工况证据。
6. 阶段顺序固定为：分析清理 -> 工程化定义草案 -> 资产归档收口 -> 共同对齐顶层建模规划。
7. M4-02A、M4-03A 的历史脚本已归档至 `归档/M4_历史脚本/`，其 CSV 为冻结证据保留在 `04_仿真结果/`，不得重跑覆盖。
8. root 级 `System_Audit` 是无端口审计支撑层，当前只包含 RMS/测量、能量/热和电池包线筛查三个内部子系统；候选包线、析锂代理和局部损耗常数尚未获得参数来源准入，不能作为设备或安全结论。
9. 建模仿真研究规范参见 `AGENTS.md` 第 11 节「建模仿真研究规范」。
10. M4/M5 正式路线命名、dSVPWM 换流解释和文献边界以 `M5_文献调研与差异分析_v01.md` 第 8 节为唯一详细记录。

## 建模仿真核心纪律

1. **文件体系**：一种系统结构只有唯一一个 Simulink 模型文件；脚本不承担计算功能，仅辅助仿真运行；说明文件按规划设计（覆盖式更新）和实施记录（增量式更新）分类。
2. **控制区分**：明确区分主动控制量（命令、参数、边界条件）和响应量（电压、电流、功率、温度、账本字段）。
3. **工况三要素**：每个工况开始前必须明确（1）系统初始值、（2）具体控制要求、（3）计算求解器设置。
4. **长计算交接**：超 MCP 响应上限的仿真由 agent 提供三件套（可执行命令、预期产物、完成判据），用户触发后 agent 只读核验。
5. **并行策略**：`parsim` 脚本侧适配，默认 2 核、上限 4 核、保留串行回退，不为并行修改模型本体。
6. **结论边界**：所有结论限定为 `Reference / Platform`，不升级为设备、安全、析锂、寿命或产品级结论。
