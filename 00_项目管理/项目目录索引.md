## 使用规则

1. 先读取 Active 文件，再决定是否调用归档证据。
2. 不将归档中的模型、脚本、计划、参数和“下一步”文字视为 M3 指令。
3. M3 的顶层目标、架构、路径、里程碑和验收以 `M3_顶层建模规划_v02.md` 冻结版为准；现有 `M3_工程兼容性模型规划_v01.md` 仍为待重审基线。
4. 当前阶段不把归档或独立的 G0-G3、Precheck、RMS 脚本或固定实施顺序视为已确认的执行主线；活动 V4 内的窗口 RMS 是已审计的 M4-01 结构字段，M4-02 的 40 ms 窗口是单次仿真的受控覆盖，不等于设备参数准入。
5. `M3-PD-CONTRACT-v01` 仍不可运行；M4-02、M4-03 运行器和各自唯一 CSV 是受控的 Reference / Platform 资产，不得被当作产品配置或完整工况证据。
6. 阶段顺序固定为：分析清理 -> 工程化定义草案 -> 资产归档收口 -> 共同对齐顶层建模规划。
7. M4-00、M4-01A、M4-01B、M4-01C、M4-02A 和 M4-03A 的交付物已形成并作为证据保留；M4 未收口，M4-02B 和 M4-03B 的范围、顺序和验收以 `M4_阶段目标达成审计与收口分解_v01.md` 为准。M4-01C 的完整账本只适用于短时锁止 Reference / Platform 工况；两轮历史扫描均不解除设备、热或安全数据缺口。
8. root 级 `System_Audit` 是无端口审计支撑层，当前只包含 RMS/测量、能量/热和电池包线筛查三个内部子系统；候选包线、析锂代理和局部损耗常数尚未获得参数来源准入，不能作为设备或安全结论。
